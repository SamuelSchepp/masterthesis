\section*{Context}

Fabrik19 is a software development company, which develops mobile apps and web applications for local businesses, events and city marketing agencies.
All of the applications are produced business to business (B2B).
There are also implementations, that power digital signage devices, like customer terminals in stores or outdoor shopping streets.
These devices show context specific information about products and the environment and provide search functionality and suggestions.

The content is loaded from back-ends, which are powered by common content management systems, like Joomla or Wordpress.
The customers have access and act as editors, who manage their own content.
Some of the data is loaded through a custom proxy service, that is reprocessing data, before it gets downloaded by the apps.
The app is then able to only load images in the needed size, so it may not be way to large to load over a mobile internet access.

\section*{Problem definition}

The customers not only want to edit their data and content live, but there is the requirement, that they need to be able to create new screens, different kinds of lists, or totally new features for the app.

if the app runs on a digital signage terminal, the new app version has to be deployed to all of the devices, which can be a hard process, if there is no automatic deploy mechanism.

If the app is targeted for mobile devices, changes have to be made for both platforms, and changes have to be reviewed by Apple and Google, which may take hours to days, which means that last minutes changes are not possible.

Many solutions for faster app deployment have been evaluated by the engineering team.
Solutions like web apps (Ionic), or native apps with JavaScript bundles (React Native) have some significant downsides.

The apps have to support native capabilities like push notifications, camera, beacons and GPS.
Using these special features in a highly customization way and independent of the framework vendors implementation is not possible.
New platform features cannot be used without workarounds until they are implemented by the framework.
Problems, that are already solved are reoccurring, because of the JavaScript layer.
Object-relational operations on a SQLite database take much more time, because data has to be mapped into the JavaScript environment, instead of heaving direct memory access with the native languages.
Web views also introduce new issues with the way, the app is used.
They change the way, users travel through the view hierarchy, because they don't support native gestures and special device needs, like the display notch on newer iPhones.
While it is very easy in iOS to let the native navigation bars handle the different devices, a web app has to calculate it by hand and has to be updated if new devices are introduced.

The most substantial issue is the dependence on these frameworks and the need to agree to their terms and conditions.
The license of React prohibits the use of the framework, if the company is in a active lawsuit against Facebook or it`s affiliates.
