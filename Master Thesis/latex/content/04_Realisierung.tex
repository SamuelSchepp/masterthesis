\chapter{Implementation}
\label{chap:implementation}

In this chapter, the approaches that are described in chapter \ref{chap:concept} are demonstrated using code examples and the real reference implementation.
In the first section, the initial state of the interpreter is explained.
This is the starting point from which the re-engineering options are applied.

In the next section, the first approach is examined.
This is the option that is described in section \ref{sec:nativeapproach} and has been actively developed.

The last section will outline a possible virtual machine implementation exemplary, that was described in section \ref{sec:virtualmachineapproach}.
Since the virtual machine approach has not been implemented yet, a simple demonstrator has been design, that implements the key ideas.


\section{Initial State}
\label{sec:initialstate}

The original iOS interpreter is written in Objective-C and uses \textit{AsyncDisplayKit}\footnote{AsyncDisplayKit was renamed to Texture. \url{https://github.com/texturegroup/texture/}} as an asynchronous \ac{UI} framework.
It allows instantiation and lay-outing of view components on any thread.
This means, that load intensive tasks can be swapped to background threads.

This makes synchronization of \ac{mos} specific behavior like condition evaluation and state management harder.
Also, custom view implementation that are used very often are more challenging to build, because they have to be a subclass of \textit{ASDisplayNode}, which represents one simple view component.
In addition, the framework's future was unknown, since it has not been maintained and was licensed by Facebook.
It has no native Swift \ac{API}.
Later, the framework moved to Pinterest, where it has been maintained more regular.

Because of the high dependence to \textit{AsyncDisplayKit} and the difficult maintenance, the next iteration of \ac{mos}, which should use Swift, needs to move away from the \ac{UI} framework.
The code basis that was used to utilize the framework became obsolete because native views, that are enriched with \ac{mos} behavior, should be used.

\section{Native Approach Implementation}
\label{sec:nativeapproachimplementation}

Instead of \textit{AsyncDisplayKit}, that is described in \ref{sec:initialstate}, a approach should be used, that primarily uses the native \ac{UI} and the main thread.
The main approaches will be outlined and explained in the following sub sections.
This approach is fully implemented and is used for the actual applications that are described in section \ref{sec:company}.

\subsection{Native User Interface}

Every layer operates directly on their native representation.
That means that if changes need to be applied, the layer manipulates it's \ac{UI} component directly.
This is necessary because some of the layouting calculations are based on native \ac{UI} \acp{API}.

An example would be the size of text in a label layer.
A text layer can be configured to update it's size automatically based on the text content.
Also, a maximum line count could be defined.
Since the native text \ac{API} only supports the calculation of the text size based on a constrained size, but not based on a line count, another approach must be taken.
In this case, the text and the calculated text size is applied to the label.
After that, \textit{sizeToFit()} is called on the layer.
It's a native method, that shrinks the label based on the maximum line count.
The resulting size is then used as the layer's frame to layout the remaining layers.

Another example is the size of all sub view of a grid layer.
Since a native grid layouts views based on a collection layout definition, the resulting size can only be determined by asking the grid on the main thread.

This means, that lay-outing and conditions happen directly on the \ac{UI}.
It keeps the code client and simple configurations are run fast.
The following listing outlines a layer and it's behavior.

\begin{lstlisting}[style=swift, caption={Layer class (excerpt)}, label={lst:layerclass}]

	func apply(databaseChanges: [DatabaseChange]) -> [ContextChange]

Listing \ref{lst:layerclass} outlines an excerpt of the Layer class.
The function \textit{update(view:)} is the bottleneck with the most significant impact.
It's responsible for re-layouting it's native components.


\subsection{Concurrency}

Since changes are happening directly on the \ac{UI} elements, the main work needs to be done on the main thread.
This also means, that if a layer needs to re-layout, the superlayer needs to relayout all of it's sublayers.
That is because of layout constraints, that are tied to the content might can change every layers frame.
Another high workload is the update of sublayers based on a database query.
This feature is called \textit{InsertLayer} and it uses a layout definition and a database query, to dynamically create sublayers.
Since layers can only be instantiated in the main thread, and changes have always be applied to the native view elements, the main thread is blocked for the time.
It is possible to extract certain tasks to a background thread, but this would not follow any pattern and would lead to poorly comprehensible code.

\subsection{Testability}

Only a minor part of the implementation is testable, because most logic code is directly intertwined with \ac{UI} code.
Also, code depends on a lot of global references of app components like the current view or the database controller.
This makes dependency injection impossible.
Code that has been develop recently has been made testable by using injection patterns, which help to write white box tests.
An example are the tree conditions of a layer state.
If the condition tree is evaluated to be true, a defined state is applied.
The condition tree is a recursive declaration of conditions, that can be combined this standard bool operators.
The leafs of the tree reference various app context values like user settings or form data.
Those values can be injected to make the evaluation of the tree testable.




\section{Virtual Machine Implementation}
\label{sec:virtualmachneimplementation}

This section illustrates a possible solution and alternative to the approach of section \ref{sec:nativeapproachimplementation}.
To demonstrate the implementation, a small demonstrator has been developed, that will show the an example scenario.
There is no actual implementation of this approach.

\subsection{Base Elements}

The first level of abstraction consists of an \ac{UI} and a \ac{VM} framework.
The \ac{UI} framework uses the \ac{VM} framework.

The \ac{VM} framework is an implementation of the mos logic like nodes and layers.
It is independent to the native \ac{UI} and device specific \acp{API}.
This example is written in Swift, so the idea is, that the framework can be used on every Swift-enabled platform like macOS and iOS.
It is also possible to abstract the framework even further to allow it to run on Ubuntu.
But while macOS and iOS share a lot of useful frameworks like CoreGraphics, Swift for Ubuntu only comes with the most basic types.
The best solution is a \ac{SPM} framework, that builds also on Ubuntu.
This way, it is ensured that there is no \ac{UI} or platform code hidden in the framework.
Most non-\ac{SPM} frameworks are the ones that use platform frameworks like \ac{UI} and are only part of the \ac{UI} framework anyway.
After a short evaluation, most current dependencies that would be part of the \ac{VM} framework are \ac{SPM} compatible.
However, some of them are unable to build on Ubuntu, because they export an Objective-C \ac{API}, which is not available under Ubuntu.

As shown in figure \ref{fig:vmelements1}, the \ac{UI} framework uses the \ac{VM} frameworks and is it's delegate.
The \ac{VM} asks to display layers and to push new views.
The \ac{UI} then creates native views and provides the actual app and it's user interface.
Also, the \ac{UI} triggers \ac{VM} events like touch of layer or app life cycle events.

Delegate is an software pattern, where frameworks store a weak reference of an object.
This object implements the framework's delegate protocol and handles events that come back from the framework.
It's similar to the callback pattern on asynchronous functions.

The virtual machine now consists of the public interface and it's internal logic.
The public interface must be as thin as possible, so that most of the work can occur on the virtual machine's thread.
For the \ac{UI} to be able to display native views, the internal layer implementations also define a small public interface.
It is used to provide relevant information about the layer.
The \ac{UI} delegate will be asked to display a certain pre-rendered layout, which consists of layers.
Rendered in this context means, that all the work of building the full hierarchy is done.
This includes layouting and evaluating \textit{GalleryLayers}, \textit{GridLayers} and \textit{InsertLayers}.
The \ac{UI} is now responsible for initializing the needed \ac{UI} elements.
This means, that for every Layer, there is a corresponding LayerView, that is provided by the \ac{UI} framework.

This separation of layer and view has additional positive side effects.
One example is the GalleryLayer, which displays it's sublayers as carousel with horizontal scrolling.
It can be configured to enable infinite scrolling, which means that the gallery does not end at the last layer, but continues scrolling to the first layer.
Since the scroll animation is continuous, the only solution has been to duplicate the sublayers internally and place them at the end of the gallery.
If the user now scrolls across the boundaries, the gallery offset is moved back by its original layer count.

	\label{fig:gallerylayercopy}
\end{figure}

Figure \ref{fig:gallerylayercopy} demonstrates this behavior.
The initial state is that the UI windows is positioned at the first copy of the first layer \textit{\#0 (copy 1)}.
If the user now scrolls past the end and is already seeing the second copies (position 1), the window is instantly moved back by exactly the width of the original layers (position 2).
This way, the user cannot tell if the offset was changed and it feels like the gallery is able to scroll infinitely.

It is possible to achieve this with only one copy, but the iOS scroll view component will decelerate the scroll animation to fast if is realizes, that there are now more views to come.
To have more buffer and therefore a soft user interface, two copies have to be made, which results in the layers existing three times.

The problem now is, that because of the duplication of layers, it is undefined which layer responds to actions that reference the layer by its name.
The virtual machine approach solves this problem, because the duplication of the layers will be done by the view.
This way, there is no layer duplication and the changes of the separate layers will be flushed to all of the native observers.

\subsection{Change Management}
\label{sec:changemanagement}

The most important key element of this alternative approach is the concept of Dirt and Flushing.
Dirt represents changes that happen to a layer a will need to be applied to the native view.
Every Layer stores a set, which is filled with dirt instances.
This makes a layer \textit{dirty}.
On well defined occasions that are explained later, a flush is executed.
A flush will take the dirt of every layer and send it to the native views.
The native view will then apply the changes that are represented by the dirt.
The dirt set will then be set to empty.

Two important conditions must be met for this concept to be useful.
The storage of the dirt must be a set, so that duplicates are not possible.
This ensures that changes are not applied multiple times.

The second condition is the frequency of which a flush is executed.
Similar to the dirt management of Angular and Zone.js\footnote{See section \ref{sec:virtualviewhierarchy}.}, changes can only occur on special, well defined occasions.

Table \ref{fig:mosactionsthatneedflushafterwards} lists all the interactions that can produce dirt.
Everything that happens in the \ac{VM} originates from one of those interactions, so only after those, one flush must be executed.
That means, that the most complex action configurations that change a lot of things results in only one flush so that the work on the main thread is as minimal as possible.
Non-Layer changes like push and pop of views are not part of the dirt system.
Instead, they are send after the flush, af special app events.
This optimizes main thread performance, because only fully evaluated layouts should be send to the \ac{UI}.

A flush is executed recursive, starting the root layer.
This ensures, that layers, that are not part of the hierarchy anymore are not triggered, so that the \ac{UI} is not performing work on layers, that will be removed anyway.

\begin{lstlisting}[style=swift, caption={Key data structures of dirt and flush}, label={lst:dirtnflush}]
enum LayerDirt {
	case frame
}

enum TextLayerDirt {
	case value
}

public class Layer {
	var layerDirt: Set<LayerDirt>
	
	public var frame: CGRect
	public var onFrameChanged: ((Layer) -> Void)?
	
	func updateFrame(to newFrame: CGRect) {
		self.frame = newFrame
		layerDirt.insert(.frame)
	}
	
	func flushDirt() -> Void {
		layerDirt = layerDirt.filter {
			switch $0 {
			case .frame:
				onFrameChanged?(self)
				return false
			}
		}
	}
}

public class TextLayer: Layer {
	var textLayerDirt: Set<TextLayerDirt>

	public var value: String
	public var onValueChanged: ((TextLayer) -> Void)?
	
	func updateValue(to newValue: String) {
		self.value = newValue
		textLayerDirt.insert(.value)
	}
	
	override func flushDirt() {
		super.flushDirt()
		textLayerDirt = textLayerDirt.filter {
			switch $0 {
			case .value:
				onValueChanged?(self)
				return false
			}
		}
	}
}
\end{lstlisting}

Listing \ref{lst:dirtnflush} shows an excerpt of a demo implementation.
The layers each have a set of their own relevant changes (= dirt).
Notice that a developer must not forget to add the dirt to the set in case something changed.
The functions \textit{updateFrame} on the Layer and \textit{updateValue} on the TextLayer make sure to add the dirt.

A native layer view registers itself for changes using the \textit{onFrameChanged} or \textit{onValueChanged}.
So if the \ac{VM} system decides that it is time to flush, the layers \textit{flushDirt} functions are called.
The TextLayer makes sure to flush it's own and it's superclass's changes.
The challenge is, to not forget to add dirt to the set and to represent the change correctly, so no change gets lost.

The same mechanism can be used to handle internal changes as well.
To make sure, that layers who update based on changed database tables do not update multiple time unnecessarily, an internal dirt flag for every database table can be used.
This way, an interaction first triggers the internal evaluation mechanism, then the internal dirt flush and the external dirt flush after that.
Examples for layers that depend on a database query are the InsertLayer, GalleryLayer, SmartListLayer and the LayoutLayer.


\subsection{Concurrency}

Since the \ac{VM} and the \ac{UI} are separate systems on separate threads, they have to be synchronized.
This must be done in a way, that the \ac{UI} is not blocked, but events and changes occur in serial.
A possible collision are the public fields of the layers.
Those are used by the \ac{UI} to initialize and change it's views.
If new changes occur right after a flush, while the \ac{UI} is still applying changes, the public fields could be written and read at the same time.

A possible solution for this is a additional state object, that is only written to right before the flush.
This makes it possible to use observables, so no custom delegates or listener mechanism has to be developed.
To implement this, \textit{ReactiveSwift}\footnote{\url{https://github.com/ReactiveCocoa/ReactiveSwift}} and it's observable property feature can be used.
So instead of having a property and it's change handler in addition to the layers actual internal property, there are only the internal and the public one.
A flush means, that for every dirt flag, the corresponding observable property's value is overwritten with the internal one.
A major advantages are the operators, that enable structured observation and mapping of values.
It also allows filtering of equal values via \textit{skipRepeats}, to improve \ac{UI} performance even more.
The \textit{skipRepeats} operator cannot replace the dirt flag system, but extend it.
The dirt system can relieve pressure on the \textit{skipRepeats} operator, because checking for equality on the sub-layer list for example might take very long, so it is better to mark the sub-layers as dirty if re-calculated so it can be overwritten only if actually changed.

The core idea to handle changes is that the \ac{UI} calls the \ac{VM}'s \acp{API} using \textit{fire and forget}\footnote{A call to the other system that returns immediately.}.
But the \ac{VM} uses synchronization to wait until the main thread is done flushing.
This way, the \ac{VM} is always blocked while the \ac{UI} is working.

As a result, the \ac{UI} is always responsive, except for the time it is applying changes.
Also, changes are applied in the correct order, and always after one another.

\subsubsection{Implemented Features}

The virtual machine is designed to support 2 layers.
A TextLayer which has a value and a ColorLayer, which has no properties.
Both Layer are inherited from a base Layer, that can be configured with a name, a layout frame, a form-data key and a list of actions.

A form is implemented, which are used by the layers to synchronize their values based on the form-data key.
This means, that the layers share their value with layers that have the same form-data key.
This logic is implemented in the virtual machine.

Each layer can have a list of actions, that are fired if the user taps on the layer.
Currently there are two actions implemented.
The GrowAction let's a layer's frame grow to the right.
The ToggleTextAction will toggle the value of a layer between ''lorem ipsum'' and ''hallo welt''.

As described earlier in section \ref{sec:changemanagement}, the dirt and flush mechanisms are implemented.
The public \ac{API} of the layers make sure, that if the user taps on the representing view, the notified layer will call the virtual machine to perform the actions.
After that, all layers are asked to flush their changes which will then be applied to the view.

A centralized app instance object, which will always be passed to layers and actions holds all of the relevant components of a virtual machine instance.
In this implementation, the instance object has the form object, the dispatch queue and the current layer list, that are currently displayed.
A dispatch queue is an abstraction of a thread, which is used to run all the machine logic.


\subsubsection{Example Configuration}

To demonstrate the approach, a demo app is configured, by letting the virtual machine create layers on it's own, which are then displayed.
Three layers are configured.
A color layer with a size of 100 by 100 pixels, a TextLayer that displays the text ''Click me multiple times'' and another TextLayer that shows the text ''... or me''.
Both text layers are configured to have the same form data key, so both layers should display the same value.
In mos it is undefined, which layer ''wins'', but in this implementation, the first one sets the form value and the second has to adopt.
The first TextLayer is configured with three ToggleText actions and one GrowAction.
The GrowAction is configured with a the layer name of the ColorLayer.
The GrowAction is implemented to find the layer who's frame will be extended by a layer name.
The ToggleText action is configured three time to simulate multiple changes.
The result should be the same as if the action would only be fired one time.
The hard-coded configuration is depicted in listing \ref{lst:democonfig}.

\subsection{Testability}

This approach allows black box testing, which is not possible with the current implementation.
A black box test of the \ac{VM} can launch an virtual app instance, tap layers and insert values into a text field or any other user interaction.
All this happens by using the public \ac{VM} interface, that would otherwise be used by the actual \ac{UI} implementation.
Those tests are platform independent and can run very fast on a Mac or inside a iOS host application.
It is now possible to hard-code views, which then can be tested by simulating user input.
This includes all of the special mos logic.

White box unit tests are also possible, since there are no public fields, but instead a app instance is passed to the sub components.
Also, platform components like network or file system have to register to the \ac{VM} by design.
This makes injecting test implementations of those components straight forward.
It is now possible to simulate an app and it's user input fully on any host system.

An example black box test of the demo app is outlined in listing \ref{lst:unittestblackbox}.
The test simulates one tap on the first text layer.
Notice, that the \ac{VM} implementation differs slightly from the previous code snippets, because \textit{ReactiveSwift} was used.
The core structure is still intact and the approach is implemented accordingly.

The test expects the color layer to increase it's width by 30 points, because of the \textit{Grow} action.
Also, it is expected that the two text layers will both have the value ''lorem ipsum'', because of the \textit{ToggleText} action and the shared form data key.

The lines 2 to 26 setup the demo app.
\textit{XCTestExpectation} is a semaphore, that can be fulfilled.
The test execution can be halted until an expectation is fulfilled by using \textit{wait(for:timeout:)}.
This way, the test host can wait for the \ac{VM} to push it's first view.
The simulation of the tap is executed in line 29.
It is a call, that would normally be done by the native layer view of the \ac{UI} implementation.
Since layers are abstracted, anyone can tap a layer, even a test host.
After the tap, the test hosts waits for the color layer to change it's frame.
The test host uses the same interface, that would be used to update the native views frame.
But instead, the host uses it to wait for the value to change, so the value can be checked.
Finally, starting from line 34, the test host checks the color layer's frame and the values of the text layers.

\begin{lstlisting}[style=swift, caption={Black box test of the test app configuration}, label={lst:unittestblackbox}]
func testApp() {
	// Prepare waiting for first view to be pushed
	let firstViewExpectation = XCTestExpectation()
	let vmProxy = VirtualMachineDelegateProxy(firstViewExpectation: firstViewExpectation)
	
	// Start the VM
	let virtualMachine = VirtualMachine(delegate: vmProxy)
	virtualMachine.boot(appConfig: AppConfig())
	
	// Wait for first view to be pushed
	wait(for: [firstViewExpectation], timeout: 30)
	
	// Check layer count of first view
	XCTAssertEqual(vmProxy.firstView?.layerInterfaces.count, 3)
	
	// Check frame of the color layer
	XCTAssertEqual(
		(vmProxy.firstView?.layerInterfaces[0] as? ColorLayerInterface)?.frameProperty.value,

\subsection{Core Rules}

To prevent coding errors or inconsistencies, there are some rules that are arising out of it.
Especially new features or changes must follow these rules, so the performance and maintainability gains are not lost.


\begin{enumerate}
	\item \textbf{One user interaction must result in only one flush.}
	
	Since this rule represents the key elements of the re-engineering approach, it is the most important one.
	There are a few exceptions in which there is \ac{UI} functionality called right before a flush.
	En example is a pushNode action, which would push a new view of layers to the stack.
	After the layers are displayed, the remaining dirt is flushed.
	A pop of a view would result in first removing the view from the stack and then flushing changes.
	The changes of layers that don't exist anymore will not be flushed, because the flushing is done recursively and the layer's are not part of the tree anymore.
	
	\item \textbf{No \ac{UI} code is allowed inside the virtual machine module.}
	
	The virtual machine code should be able to compile on any Apple platform.
	The gold standard would be a platform independence, that allows the virtual machine code to build on Ubuntu.
	As a result, importing UIKit, CoreGraphics, CoreLocation, UserNotifications, EventKit, AVFoundation, MapKit and other modules are is not allowed.
	Only Foundation and other cross platform modules should be used.
	
	\item \textbf{The callee of a public interface must make sure to continue on the correct dispatch queue}
	
	This helps the code to be more understandable, since random queue changes are harder to track.
	With this rule, a public interface can be used without the need to check the queue requirements.
	Queues should only be switched while crossing framework borders.
	
	\item \textbf{Changes to internal layers that are relevant to the \ac{UI} must result in the appropriate dirt flag to be set.}
	
	It is important for the internal logic implementation to strictly set the according flags on changes.
	The lay-outer must set the frame flag, but only if the layout frame actually changed.
	Sub-layered layers must set their sublayer flag, if they decide to reload their layer on a database change.
	Those are some examples of code positions, that need to be disciplined.
	
	\item \textbf{\ac{VM} \acp{API} must be called using fire and forget.}
	
	To ensure a responsive \ac{UI}, the \ac{VM} must make sure to switch on the \ac{VM} dispatch queue asynchronous.
	Since every callee must make sure to switch to the correct queue, the \ac{VM} is responsible to perform asynchronous.
	
	\item \textbf{\ac{UI} \acp{API} must be called synchronous, so the \ac{VM} waits for the \ac{UI} to finish.}
	
	The \ac{UI} make sure to respond synchronous, to make sure that it is safe to read the interface fields to update the \ac{UI}.
	If the \ac{UI} is asked to update some layers and some asynchronous task inside the \ac{VM} is triggered that will write those properties, a thread collision is possible.
	A solution is to let the \ac{VM} wait until the \ac{UI} is done updating the layer.
	
	\item \textbf{The public interface of the \ac{VM} must be as thin as possible.}
	
	To prevent programming errors and inconsistencies, every public interface must be evaluated if it complies with the \ac{API} contract, like the rule that results of interaction are applied by a flush.
	The \ac{VM} must perform as much work as possible and delivers pre-rendered data to the \ac{UI}.
	Pre-rendered data is data that has already been evaluated and is ready to be displayed.
	
	\item \textbf{Platform interfaces are provided by the \ac{UI} framework, but are run on the \ac{VM}'s dispatch queue.}
	
	For the \ac{VM} to be fully testable, every interface to platform specific functionality like the file system, database, user defaults, screen size, location, network and push notifications are provided by delegates, that are registered in the \ac{VM}.
	This way, the \ac{VM} module can be fully black-box tested by unit tests.
	It is also possible to port the application to any platform.
	To prevent \ac{UI} thread utilization, the interfaces are provided by the \ac{UI} framework, but run on either the \ac{VM} queue or a custom background queue.
	This way, tasks like querying the database does not block the main thread.
	
	\item \textbf{No singletons or static objects.}
	
	To keep the code testable, no singletons or public static objects should be used.
	Instead, a app instance object, that represents one app instance should be passed to every sub-component.
	This makes it possible to run any amount of virtual apps in parallel, to enable fast testing.
	Also, it allows for dependency injection, to perform white box unit tests.
	
	
	
\end{enumerate}

\subsection{Maintainability}

To make code maintainable and abstract the different logic levels, like configuration parsing, dynamic logic and interaction, some implementation rules must be set.
The current implementation disregards all of the rules to some extend.

\begin{enumerate}
	
\item \textbf{Every component of the \ac{JSON} model must be represented by a fully typed and read-only data structure.}

To prevent uncoordinated side-effects and a safe interface to the model, the text based model must be represented by a read-only data structure.
This guarantees a strict separation of parsing logic, dynamic behavior and side effects.
If a field of the model is considered mutable, a additional field is added in one of the dynamic logic classes, but not in the representation structure.
This ensures that the time of applying the model to the mutable data is strictly defined and the original configuration is still in tact.

\item \textbf{Every interface between virtual machine and \ac{UI} must be defined by a protocol\footnote{A protocol in Swift is similar to an interface in Java. It also allows the definition of public properties.}.}

Any implicit class \ac{API} lacks comprehensibility, if it is spread across the class definition.
To keep the interface clear, it should be defined as a custom protocol, so the intention and usage are explicit.

\item \textbf{Class dependencies must be defined by a protocol.}

If a class depends on another class's functionality, a protocol should be defined, that an handle the classes requirements.
This is important for test purposes, since a white box unit test must be able to simulate external behavior.
The test host can pass a custom object, that implements the protocol, but simulates special behavior and values, to test the border cases of the class.

\end{enumerate}
