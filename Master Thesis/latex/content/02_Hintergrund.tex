\chapter{Background}

The system, that is discussed here, is a model interpreter that is designed to run mobile applications on iOS and Android.
The system, which includes the interpreters and the backends are called \ac{mos}.
It is in active development and powers over 50 apps that all are available in the Apple AppStore and Google PlayStore.
The app categories range from retail to event to city marketing apps\footnote{\url{https://fabrik19.de/technologie/}}.

The creator of the system is Fabrik 19 GmbH.
The company is based in Gießen, Hesse, Germany and was founded 2013 by Mark Pralle and Sven Herchenhein.
At the moment, the company has bout 25 employees.
Besides of apps, there are also other projects and solutions, like consulting and web hosting.

Other stele systems are also active in Globus hardware stores and Globus grocery stores.

\section{Model Interpreter System}

The main development focus lays on the app interpreter platform ''mos''.
It supports the two major platforms iOS and Android, but there is also a lightweight version available in .Net, which is running on the steles as a Windows app.
A web client is also actively in development, which aims to replace the .Net version.

A configuration are multiple JSON files, that represent the app meta information, its layout and its behavior.
The files are called node.
There is one root node, whose url is baked into the app while compiling.
This root node defines the first screen of the app.
All the other screens can be defined in other nodes.
When displaying another node, the new node is loaded from cache or network and its new view is pushed on the view stack, which is done by the interpreter.
On the particular platforms, this action will look different and will use the platforms native animations.
Web apps like Ionic have their own push animations which makes them feel foreign.
The previous node can pass data, like the ID of the item that should be displayed.

The goal is, to be able to create a new app in one day, by meeting with the customer and configuring the app together.
The syntax of the configuration is designed to abstract all of an apps standard behavior.
Because of a custom layout system, that can be described with constraints and will layout automatically, it is not possible to recognize that it is only a configured app, instead of a custom native one.

The apps have sophisticated caching and optimization mechanisms, which makes it possible to use them completely offline, without any additional configuration effort.
The configuration, images and data are cached heavily, which makes the apps very fast.
But because of the dynamic design of the configuration syntax, it is possible to clear the caches and force new versions.
The cache first policy of the interpreter makes sure, that contents are always available and loads changes in the background.

Besides of displaying data, the client is highly intractable and can perform various custom tasks on the device, like uploading local data, saving data to the devices user settings and providing push, beacon and location infrastructure.

This data can be used in describing the layout by defining layer conditions, which makes the app behave accordingly.


\subsection{Use Cases}

The main use cases are marketing and customer interaction.
The concept is data source driven and enables configuration of any data source.
Because of a template engine, data can be arranged for display in an intuitive way.

Since the configuration can be changed on the fly and without the need of providing an update via app store, the contents can be adjusted to fit current events like holidays or promotions.
Especially for digital signage it is important to be able to change the contents without any application updates.

Any change to the configuration is reflected as soon as the app checks for configuration changes.
The update mechanism are also highly customizable, so that the optimal update intervals and strategies can be used for different apps.

A city marketing app might need a medium update interval, so that promotions and special content are available when needed.
A event app should have a very short update interval, so that the events program changes or announcements are delivered as fast as possible.


\subsection{Interoperability}

The app can be filled with data via common REST services, that deliver JSON data.
The app uses HTTP mechanisms and a custom delta system to only load new data.
The mos backend system also has a data proxy, which can load any data and provide it in a simpler, preprocessed format, so it is simple for the apps to download.
This way, it is possible to use any data source and still be able to change the templates and layouts without updates.

The apps have the capability to be integrated in a Wordpress instance, a chat service like Mattermost or an secured \ac{API} that uses authentication methods like client certificates, authorization headers or JSON web tokens.

By default any network communication is secured with an app specific client certificate.
Other authentication methods have to be configured.

\section{Models}

Since the mos-Framework tries to abstract mobile application platforms into a common language, the configuration could be considered to be an abstract model of a mobile application.
To classify and evaluate the model approach for this domain, the following section will discuss models and their properties.

\subsection{General Model Theory}

A general model is described by Herbert Stachowiak\footnote{\cite{stachowiak_allgemeine_1973} 131}.
He defines three properties, that apply to all models.

\textbf{Abbildungsmerkmal} (Representation): A model is a representation of an original, that is either a natural or artificial thing, that can have properties and behavior.
The original can be a model it's self.

\textbf{Verkürzungsmerkmal} (Reduction): The second property is the ability to reduce properties and behavior of the original to a set, that is relevant for the model handlers.
The original might be a complex system and only a subsystem or specific properties and their interaction are relevant.
For this, the model can reduce the complexity, to be more simple to understand and to minimize the error rate.

\textbf{Pragmatisches Merkmal} (Pragmatism): Models can not be clearly assigned to their originals.
That means, that original and model are not equal, but can be interchanged.
There are three cases defined by Stachowiak, in which the model can be used as an replacement.
Models are not only models of something, but they are also models for someone, a person or an artificial model user.
They perform their functions within a time interval.
And they are models for a specific purpose.
Often times, the purpose is an action that is not possible on the original because it may be to complex, or an physical original could be destroyed.

If theses properties apply an object, i can be called model according to Stachowiak.
It can be applied by numerous of common usages of the word.
Mathematical models, philosophical models, simulations and concrete object models are some examples, that fit the requirements.

\subsection{Specialized Model Theory}

To classify models in a more specific way, they can be categorized as descriptive and prescriptive models.
A descriptive model represents an existing system, as it describes it's current behavior.
The general model specifications can still be applied, since all three properties are true for such a model.
It cannot be modified unless the original changes, since it is representing it directly.
A change to one of them without updating the other would result in the model not representing the original, which makes it a prescriptive model.
A change on the original which is not relevant to the model will keep its validity.

Models, that represent a fictive object or system are called prescriptive models.
The prefix ''pre-'' indicates, that the model exists before the original.
The model is then used to prototype a system, and test their behavior and properties, without the need to create the full system.

According to Seidewitz 2003 \cite{seidewitz_what_2003} both descriptive and prescriptive models define a set of originals.
That means, that a original can be derived from a model by adding additional information that does not contradict each other.
Often times, the information that must be added is trivial or can be fetched from somewhere else.
That is possible, because the model has by definition all of the necessary information.

A derived original fulfills all of the models properties, otherwise the original is not a correct derivation.

\section{Model Driven Development}

Models can be used in a software development process (\ac{MDSD}) or as part of a software architecture (\ac{MDA}).
Software is also a model, which represents real life objects and procedures.
Since it is well structured by default, it is fairly easy to create model from a existing code basis.
\ac{UML} is a widely used modeling languages, which can represent static, dynamic and interactive aspects of software systems.

The problem is, that the domain specific structure of the code can be hard to understand, because the abstrcation level of a modern object oriented programming language is much lower, than the domain requires.
\MakeUppercase{Völter} and \MakeUppercase{Stahl} describe the result of scattering relevant information in the code as:
\textquote{The much-treasured inner structure is present in a cloudy, distributed, and of course also a strongly individualized form.}\footnote{\cite{stahl_model-driven_2006} 14}
That means, that the domain's structure, components and behavior definition is represented in the software code, but in a way that is hard to understand.
That is, because the relevant parts and the domain critical definitions are spread in the software's code.
It is also harder to unterstand a domains properties, by only reading the programming language.

A software model, like an \ac{UML} class diagram are an old tool to document and model software systems.
The problem here is, that the abstraction level of an \ac{UML} class diagram is often times the same as the programming language itself.
A developer could find himself transforming a class into a diagram or generating a diagram, which does not contribute to the understanding of the domain structure.
This happens, because the programming language itself is abstracted, but not the domain itself.

To address the problem of widely spread domain implementations, the programs important and relevant structure is compressed in a model, which defines and represents the program on a abstract and formal level.
The model can be a textual \ac{DSL}, a graphical diagram or even an informal text, that is then converted into a concrete syntax.
It contains all the domain relevant information, which is required to derive a full implementation from it.

This procedure results in a model becoming part of the source of a program, and not only the documentation.
A good model syntax can lead to the model being the documentation itself.

\subsection{Modeling a Software System}

An existing implementation can be analyzed, to create a model from it.
If no model infrastructure exists, this is the way to approach the transformation.

Figure \ref{fig:mdsd1} shows, how the code basis of an existing implementation of a software system can be split into three categories\footnote{\cite{stahl_model-driven_2006} 15}.

The \textbf{schematic repetitive code} is the part, that needs to be abstracted into a model.
It contains the application specific logic, that is later modeled by a user.
It does not contain code, that is equal for all applications in the same domain.

\textbf{Generic code} is derived from the domain and target platform and not from the concrete application.
Therefore, it is not modeled, but generated by the domain's \ac{MDD} infrastructure.
It is mostly boilerplate code, that should not be part of the model.
It is the same for every application and is later part of the generator templates.

\textbf{Individual code} is unable to be represented in the model, because it is to specific and does not fit into the domain.
The tools need to make sure that this code can be integrated into the generated code later.
To simplify the \ac{MDSD} process, individual code should be avoided as far as possible, because in the worst case the individual code makes up all of the application code, which eliminates the possibility to use a model to represent the application.
In the best case, the individual code portion is empty.
This allows for a more simpler tools, because the application can be generated completely out of the model and there is no need for individual code interfaces.

	}

Figure \ref{fig:mdsd2} shows, how the analyzed parts can now be separated into source code and steady code.
Source code describes the part of the code that is application specific and is developed by the \ac{MDD} user.
The application model and individual code should be checked into source control, since that is the actual source of the application, given a \ac{MDD} infrastructure exists.

Platform and domain code is part of the \ac{MDSD} tools.
It is baked in, so that it is part of the output while generating.
In case of code generators, this code would be part of the templates.
In theory, it is now possible to develop any app, that fits into the domain, by only defining the model and the individual code.
Extensions of the model's capabilities might need to be reflected in the platform or domain code.


\section {Code Generators}

Now that there is a model of an application, a concrete implementation has to be created from the model.
Code generators can take the model and generate a software system, that is an instance if the model.
Another approach are model interpreters, which execute the model directly, similar to a virtual machine.

\subsection{Meta Programming}

Code generation enables advantages, that are available to the generated code.
By using meta programming more concepts can be expressed than the target language allows.

\subsubsection{Code efficiency}

A code generator has the privilege to build highly efficient code, by using the fastest statements, which have the fewest overhead.
Modern programming language features like flexible syntax, polymorphism or reflection cause the application to run slower then needed\footnote{according to \cite{stahl_model-driven_2006} 181}.
A code generator can perform polymorphism and reflection on generation time and generate direct code, that is fast.
The flexibility now lies in the model and not in the generated code, which tends to be impossible to maintain, because of the optimization potential.
Code size can also be reduced, by only creating the needed parts.

Aspects of the applications functionality that normally would be scattered around in the code can be modeled centralized and then generated into the depended locations.
An example would be a dependency framework, that is used by many classes.
Instead of copy the import code to every class, it can be modeled as an decorator, that will provide the needed boilerplate code to all classes.

\subsubsection{Platform language restrictions}

Is is possible to break out of the target language's restrictions.
The level of abstraction of the meta model and its defined language does not need to match that of the target language.
A code generator can generate code that reflects higher level of features.
It is possible to generate code that integrates the concept of \ac{OOP} into a non \ac{OOP}-language.
For example: A generator can generate structs and function pointers in a C/C++ application to imitate objects.
The generated code on it's own is not considered safe or structured, but the static context analysis what is done while creating the model performs the sames checks as an \ac{OOP}-compiler.
Another example are generics, that can by used generating untyped objects\footnote{$Object$ (Java), $void*$ (C/C++), $any$ (JavaScript)} and perform a static context analysis while generating the code.
Any feature that is known from modern languages can be used by imitating the compiler and its static context analysis.
In a more extreme case, a virtual machine can be integrated into the application which can run any code.

Generated code with flat hierarchies is easy to analyze statically, because much of the logic is not abstracted into frameworks.

\subsubsection{Compatibility}

The higher the abstraction level of the meta model, the more platforms can be supported.
The static context analysis is independent to the platform and should not integrate platform specific restrictions.
A transition to another platform or supporting multiple platforms at once is possible and is one of the core strengths of \ac{MDD}.

\subsection{Generation}
\label{sec:generation}

Now that a model exists, code can be generated out of it.
Before code can be generated, the model has to be transformed into a generation oriented model.
It's purpose is the simplification of the generation process by providing the target structure, which then can be converted to text directly.
The transformed model is an model instance, that is targeting a specific platform.
In cases of which the model's abstraction level is low and close to the target platforms, the transformation is not needed, since it brings a new level of complexity and abstraction to the process.

A typical example is a high level model, which could be implemented in any imperative language would be transformed into a \ac{UML} class model.
Object oriented programming code can then be generated out of the class model trivially.
There are techniques to create program code from a transformed model.
The two most popular ones\footnote{according to \cite{stahl_model-driven_2006} 186} are templates- and \ac{API}-based code generators.

\subsubsection{Template Based Generators}

A template is a definition, that is iterating over the model and prints the values on the corresponding locations in a text file.
A template mixes the concrete target code presets with a markup notation, that is getting and writing values.
The handlebar notation is one of the simplest form of template based code generation.
A well formed model, which is considered the input is applied to a template, which results into a resulting output.

The following example tries to generate a HTML file with an overview of all used licenses of a mobile application, by using \ac{XSLT}.
The model, which is created by a human user, contains only necessary information.
Listing \ref{lst:licencesmodel} shows an example, that consists of three dependencies, two of which use an MIT and the other an Apache license.

\begin{lstlisting}[caption={licences.xml - The model that is representing a licences web page}, language=html, label=lst:licencesmodel]
<licences>
	<carthage>
		<name>Alamofire</name>
		<url>https://github.com/Alamofire/Alamofire</url>
		<id>MIT</id>
		<copyright>Copyright (c) 2014-2018 Alamofire Software Foundation (http://alamofire.org/)</copyright>
	</carthage>
	<carthage>
		<name>Changeset</name>
		<url>https://github.com/osteslag/Changeset</url>
		<id>MIT</id>
		<copyright>Copyright (c) 2015-18 Joachim Bondo</copyright>
	</carthage>
	<carthage>
		<name>Couchbase Lite</name>
		<url>https://github.com/couchbase/couchbase-lite-ios</url>
		<id>Apache</id>
		<copyright>Version 2.0, January 2004 http://www.apache.org/licenses/</copyright>
	</carthage>
</licences>
\end{lstlisting}

Now, a template and processor needs to create a \ac{HTML} file, that displays the dependencies and a copy of their respective license.

\begin{lstlisting}[caption={template.xslt - An XML file, that is later tranformed to XHTML}, language=html, label=lst:licensetemplate]
<xsl:stylesheet version="1.0" xmlns:xsl="http://www.w3.org/1999/XSL/Transform">
	<xsl:template match="/licences">
		<hr />
		<h1>iOS</h1>
		<hr />
		<xsl:apply-templates select="carthage"/>
	</xsl:template>
	<xsl:template match="carthage">
		<h3><xsl:value-of select="name" /></h3>
		<a>
		<xsl:attribute name="href"><xsl:value-of select="url" /></xsl:attribute>
		<xsl:value-of select="url" />
		</a>
		<h4><xsl:value-of select="id" /> License</h4>
		<p><xsl:value-of select="copyright" /></p>
		<xsl:if test="id = 'MIT'">
			<p>Permission is hereby granted ...</p>
		</xsl:if>
		<xsl:if test="id = 'Apache'">
			<p>TERMS AND CONDITIONS ...</p>
		</xsl:if>
		<hr />
	</xsl:template>
</xsl:stylesheet>
\end{lstlisting}

Listing \ref{lst:licensetemplate} reveals some major advantages:

\begin{enumerate}
	\item Changing the style and layout of the HTML file requires only a few changes. Adding or changing the \ac{CSS} classes of elements only requires to change the one occurrence in the template.
	\item The \ac{URL} of an entry was displayed as the text content of an $a$ element, but also occurs in the $href$ attribute. Changing the \ac{URL} later, can cause inconsistencies, if one of the two occurrences are omitted.
	\item Instead of copying the license text to each entry, only the identifier (e.g. $MIT$) has to be defined.
	The template appends the correct text automatically.
	\item Defining a template and a model of this size actually made the handling more easy and future changes can be made much faster, while keeping the file consistent.
\end{enumerate}

Since \ac{XSLT} was created for web applications, \ac{PHP} offers native support and an \ac{XSLT} processor.
To generate the HTML file, the script in listing \ref{lst:phpxsltgenerate} can be ran.

\begin{lstlisting}[caption={generate.php -  Generation script}, language=php, label=lst:phpxsltgenerate]
<?php
// Input
$xml = new DOMDocument;
$xml->load('licences.xml');

// Template
$xsl = new DOMDocument;
$xsl->load('template.xslt');

// Transformer
$proc = new XSLTProcessor;
$proc->importStyleSheet($xsl);
$output = $proc->transformToXML($xml);

// Write to file
$file = fopen("./LICENCES.html", "w") or die("Unable to open file!");
fwrite($file, $output);
fclose($file);
?>
\end{lstlisting}

Figure \ref{fig:licences_html} displays the resulting \ac{HTML} file.
This file can now be bundled in to the app and enriched with CSS and be displayed in the apps license menu.
The generation of the file can also be automated, by including the generation in the application's continues deployment pipeline.
The model, the template and the generation script would be committed into version control.
Another approach would bundle the template and the script into an external tool, which will be developed and maintained separately.
In that case, only the model would be part of the version control and thus the application's source.

In larger projects, the template can become very complex.
Using intermediate models as described in section \ref{sec:generation} that are generated from the source model, can reduce the complexity, branching and nesting, because the intermediate model has either the same or a correspondingly pleasant structure.
Nevertheless, templates will get very big very fast and are bound to the structure of the target platform.
In \cite{stahl_model-driven_2006} this approach is called ''Templates and Metamodel'', while \textit{Metamodel} is the intermediate model, that is used to generate code directly.

\subsubsection{API-based Generators}

The downside of templates are their untyped and unsafe character.
It is possible to make small syntactic errors, that are unsafe and could cause the result to fail.
Especially in interpreted target languages like \ac{PHP} or \ac{HTML} the error only occurs when the respective line is parsed or ran.
Since the template elements are scattered in the file and important characters are hard to make out, this approach might be unsuitable for large projects.

An \ac{API}-based generator is a programming library for a development language, which provides an interface to the \ac{AST}.
It's purpose is to build up an \ac{AST} using a type safe language like Java.
Now the generator can be structured with Java's language features like Polymorphism, Interfaces and inheritance.
The \ac{API} makes sure, that only syntactically correct code is produced, because the \ac{AST} does not allow it.

It is also possible to generate code for different target languages, which use the same \ac{AST}.
For example, it is possible to generate Kotlin instead of Java code, since Java is a subset of Kotlin.
Also, the .NET runtime makes it possible to generate Visual Basic, C\# and C++ code from one \ac{AST}.

\begin{lstlisting}[caption={Using an \ac{API} to generate Java code, written in Swift}, language=java, label=lst:javagenerator]
let studentClass = ClassDeclaration(named: "Student");
studentClass.baseType = personClass
studentClass.typeAttributes = .public
\end{lstlisting}

\begin{lstlisting}[caption={Java code result of lising \ref{lst:javagenerator}}, language=java, label=lst:generatredjava]
public class Student extends Person {
}
\end{lstlisting}

Listing \ref{lst:javagenerator} shows an example usage of an \ac{AST}-based Java code generator, which is written in Java itself.
This form of meta programming allows the creation of any valid code by using an object oriented interface.
Since it is independent from any textual representation of the language, syntax errors in the generated code are impossible and any other language with an compatible \ac{AST} can be generated.

The \ac{AST} framework could make use of reflection to provide a more safe interface.
For example, it could be possible to reference existing Java classes not by it's string name, but rather by using direct type references, like $typeof (System.Object)$.
It might limit the set of compatible target languages, but in the example of Java, Kotlin is still compatible.

Using jQuery to manipulate or build \ac{HTML} content is also a form of \ac{API}-based code generators, since jQuery provides a safe\footnote{Not all jQuery programming errors can be checked on compile time, despite a typed interface using TypeScript. Syntax errors in the generated code are not possible.} interface to the \ac{DOM}. The generated \ac{HTML} code can then be read from the jQuery \ac{AST} object.

That means, that existing solutions\footnote{E.g. CodeDOM for .NET Applications \url{https://docs.microsoft.com/en-us/dotnet/framework/reflection-and-codedom/using-the-codedom}} can be used.

This approach is suitable for large projects, that need a lot of generated code.
If most of the generated code is boilerplate code, which has only few injected lines, templates might be the better approach.
It is possible to mix these approaches by generating parts independently.
