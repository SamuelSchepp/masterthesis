\chapter{Background}
\label{chap:background}

The system, that is discussed here, is a model interpreter that is designed to run mobile applications on iOS and Android.
The system, which includes the interpreters and the backends are called \ac{mos}.
It is in active development and powers about 65 apps that are mostly available in the Apple AppStore and Google PlayStore.
The app categories range from retail to event to city marketing apps\footnote{\url{https://fabrik19.de/technologie/}}.

\subsection{Challenges}

To be able to deploy dynamic apps and change it's contents live, a model-driven interpreter architecture is being used.
The contents and behavior are described by modeling the app as an abstract configuration.
In addition, the client application is developed to reload it's contents and configurations if changed.
This way, an application that has been deployed one time will always be up to date.
This is needed for mobile applications but more importantly for digital signage installations.
That is why, this organization chose a model-driven development and interpreter architecture approach for it's mobile applications.
This can be applied to similar situations, where the goal is to produce many application with similar structure, or where behavior and implementation has to be separated, because of complexity.

One example is the generation of C-code for embedded systems using Simulink\footnote{\url{https://www.mathworks.com/products/simulink.html}}.
It allows designing and modeling algorithms and structures by setting up graphical block diagrams.
It is possible to structure algorithms in systems and subsystems, which behave as a block themselves.
It is then possible to run simulations on those models, which may not be possible on the native code, because it must be integrated into a testable simulation environment.
The artifact is platform specific C-code, whose behavior is considered equivalent to that of the model.

Another example are video games or augmented reality applications, where there is a strict separation of content and rendering.
Content developer are able to artistically create the worlds and objects, while engine experts are available to maintain the rendering and hardware specific optimizations.

Those two examples show, that model-driven development is used in several domains and it is a possible solution to manage complexity.
Additional benefits like the ability to switch to another model without changing the app are the core ideas why the Fabrik19 organization chose this approach.

\section{Model Interpreter System}

The main development focus is on the app interpreter platform ''mos''.
It supports the two major platforms iOS and Android, but there is also a lightweight version available in .Net, which is running on the steles as a Windows app.
A web client is also in active development, which aims to replace the .Net version.

A configuration are multiple JSON files, that represent the app meta information, its layout and its behavior.
These files are called node.
There is one root node, whose \ac{URL} is baked into the app on compile time.
This root node defines the first screen of the app.
All other screens can be defined in additional nodes.
When displaying a different node, the new node is loaded from cache or network and its new view is pushed on the view stack, which is done by the interpreter.
On the particular platforms, this action will look different and will use the respective platform's native animations.
Web apps like Ionic have their own push animations which makes them feel foreign.
The previous node can pass data, like the ID of the item that should be displayed.

The goal is, to be able to create a new app in one day, by meeting with the customer and configuring the app together.
The syntax of the configuration is designed to abstract all of an apps' standard behavior.
The custom layout system of mos enables unrestricted, dynamic and relative layouts, which allows for almost any layout.
This way, it is not obvious, that the app was created using a framework.

The apps have sophisticated caching and optimization mechanisms, which makes it possible to use them completely offline, without any additional configuration effort.
The configuration, images and data are cached heavily, which makes the apps very fast.
But because of the dynamic design of the configuration syntax, it is possible to clear the caches and force the download of new versions.
The cache first policy of the interpreter makes sure, that the contents are always available and loads changes in the background.

Besides of displaying data, the client is highly intractable and can perform various custom tasks on the device, like uploading local data, saving data to user settings and providing push, beacon and location infrastructure.
This data can be used in the layout by defining layer conditions, which makes the app behave according to the data, which adds dynamics to the \ac{UI}.

\subsection{Use Cases}

The main use cases are marketing and customer interaction.
The \ac{mos} concept is data source driven and enables configuration of various data sources like Facebook news or online blogs.
A template engine arranges and combines the data for display, which makes the layout definitions independent to the data structure.

Since the configuration can be changed on the fly and without the need of providing an update via the App Store or Google Play Store, the contents can be adjusted to fit current events like holidays or promotions.
Especially for digital signage it is important to be able to change the contents without any application updates.

Any change to the configuration is reflected as soon as the app checks for configuration changes.
The update mechanism are also highly customizable, so that the optimal update intervals and strategies can be used for different apps.

A city marketing app might need a medium update interval, so that new promotions and special contents are available every day.
An event or conference app should have a very short update interval, so that program changes or announcements are delivered within minutes or as fast as possible.


\subsection{Interoperability}

The app can be filled with data via common REST services, that deliver JSON data.
The app uses HTTP mechanisms and a custom delta system to only load new data.
The mos backend system also has a data proxy, which can load any data and provide it in a simpler, pre-processed format, so it is simple for the apps to download.
This way, it is possible to use any data source and still be able to use simple data structures in the app.

The apps have the capability to be integrated into a Wordpress instance, a chat service like Mattermost or an secured \ac{API} that uses authentication methods like client certificates, authorization headers or JSON web tokens.
By default any network communication is secured with an app specific client certificate.

Multiple mobile application for several customers are in production.
Those are powered by a comprehensive infrastructure, that delivers app configurations, build systems and dynamic data endpoints.

In addition to the following components, a continuous delivery system is in development, that allows automation of uploads to the app store.

\section{Models}

Since the mos-Framework tries to abstract mobile application platforms into a common language, the configuration could be considered to be an abstract model of a mobile application.
To classify and evaluate the model approach for this domain, the following section will discuss general models and their properties.

\subsection{General Model Theory}
\label{generalmodeltheory}

A general model is described by Herbert Stachowiak\footnote{\cite{stachowiak_allgemeine_1973} 131} in 1973.
He defines three properties, that apply to all models.

\textbf{Representation characteristic}: A model is a representation of an original, that is either a natural or artificial thing, that can have properties and behavior.
The original can again be a model.

\textbf{Reduction characteristic}: The second property is the ability to reduce properties and behavior of the original to a set that is relevant for the instances that handle the model, like software or humans.
The original might be a complex system and only a subsystem or specific properties and their interaction are relevant.
For this, the model can reduce the complexity, to be more simple to understand and to prevent errors.

\textbf{Pragmatic characteristic}: Models can not be clearly assigned to their originals.
That means, that original and model are not equal, but can be substituted.
There are three cases defined by Stachowiak, in which the model can be used as an replacement.
Models are not only models of something, but they are also models for someone, a person or an artificial model user.
They perform their functions within a time interval.
And they are models for a specific purpose.
Often times, the purpose is an action that is not possible on the original because it may be to complex, or an physical original could be destroyed.
Depending on what the objective is, the original can be replaced by the model.

If theses properties apply an object, it can be called model according to Stachowiak.
It can be applied by numerous of common usages of the word.
Mathematical models, philosophical models, simulations and concrete object models are some examples, that fit the requirements.

\subsection{Specialized Model Theory}

To classify models in a more specific way, they can be categorized as descriptive and prescriptive models.
A descriptive model represents an existing system, as it describes it's current behavior.
The general model specifications can still be applied, since all three properties are true for such a model.
It cannot be modified unless the original changes, since it is representing it directly.
A change to one of them without updating the other would result in the model not representing the original, which makes it a prescriptive model.
A change to the original which is not relevant to the model will retain the model's validity.

Models that represent a fictive object or system are called prescriptive models.
The prefix ''pre-'' indicates, that the model exists before the original.
The model is then used to prototype a system, and test their behavior and properties, without the need to create the full system.

According to Seidewitz 2003 \cite{seidewitz_what_2003} both descriptive and prescriptive models define a set of originals.
That means, that an original can be derived from a model by adding additional information.
Often times, the information that must be added is trivial or can be fetched from somewhere else.
That is possible, because the model has by definition all of the necessary information.
A derived original fulfills all of the models properties, otherwise the original is not a correct derivation.

\subsection{Model Representation}

It is common to represent models either by text or using graphical editors.
Both are suitable for their respective use cases.
If it is advantageous that objects and their relationships can be arranged on a two dimensional surface, a graphical editor can be used.
If the model syntax is similar to a programming language, graphic editors might obstruct the workflow.
A text-based representation is commonly used in software development, because it is easy to integrate models into an existing infrastructure like \ac{VCS}, because those systems handle text files.
Also, tools for developing \acp{DSL} or code generators mostly operate on text based model representations, therefore source code.

Graphical models can be easier to understand at first, while the complexity may increase with growing model size.
Examples for graphical models are \ac{UML} diagrams or visual programming languages like Scratch\footnote{Educational visual programming language for children. \url{https://scratch.mit.edu/}}.

\section{Model-Driven Architecture}
\label{modeldrivenarchitecture}

To handle an evolving infrastructure, changing requirements and technology development, a model-driven architecture (\ac{MDA}) standard has been developed by the \textit{Object Management Group}\footnote{https://www.omg.org/} \cite{miller_mda_2003}.
The goals are to lower costs and reduce complexity of software maintenance.
The \ac{OMG} noticed, that the requirements of a software system is much more than only the product itself.
In addition, interoperability, changes to the environment and data migration require constant maintenance and continuous progression.
Since classic applications are written in a platform dependent programming language, those task tend to be very costly, because there is a large code basis and each application must be maintained individually.
To be able to produce software that is build to last, easy to integrate, easy to change and able to keep up with new technology, \ac{OMG} defined a standard, \ac{MDA}, on how to model a system.
Models separate the implementation details that are required to run in a certain environment from the actual way of using the platforms capabilities (\cite{miller_mda_2003} 2.1.2).
This model is in a certain level of abstraction and can be transformed to platform specific models and therefor executable code.
Now, with the model as input, parts of the application can be generated.
If requirements change, the model which is human readable can be changed and new code be generated instantly.
If the platform changes or new technology like a different data storage or programming language occurs, the tool chain can be updated and all of the generated code can be re-generated to reflect the current technology state.
This leads to long term flexibility of implementation details, integration capabilities, maintenance complexity and simulation (\cite{miller_mda_2003} 1.2).

There are three levels of abstraction defined by \ac{MDA} (\cite{miller_mda_2003} 2.2.11).

\begin{itemize}
	
	\item \textbf{Platform Specific Model (\ac{PSM})}
	
	A model defines, how platform specific functionality is used.
	An example is a \textit{Jenkinsfile}, that models a pipeline of tasks that can be run on a Jenkins build service.
	It is a model, because there is no concrete code implemented, but instead jobs are modeled by using predefined task definitions and a stage structure.
	It is platform specific, because Jenkins-specific functionality can be used like defining build status or using plugin interfaces.
	
	\item \textbf{Platform Independent Model (\ac{PIM})}
	
	The model is suitable for more than one specific platform.
	This is achieved by defining a set of functionality that is implemented by multiple platforms.
	A typical example are \ac{UML} class or activity diagrams, because they define a structure, without relying on a platform.
	Many platforms support module and class access modifier, so that public, protected and private can easily be mapped to many other platforms.
	As a consequence, \acp{PIM} can be mapped to \acp{PSM}.
	
	\item \textbf{Computation Independent Model (\ac{CIM})}
	
	Not only is this kind of model independent to a platform, but also to the way the result is achieved or calculated.
	This is typical for declarative models and allows the most freedom for the generator or interpreter.
	An \ac{SQL} statement can be considered computation and platform independent, since it can be used on multiple \ac{SQL} based platforms and it does not model the way the result is evaluated.
	Some may argue, that \ac{SQL} is to low level in some contexts and in fact does describe the way of operating on data.
	This point of view depends on the models context.
	If the \ac{SQL} statements are directly used by handwritten code, it can be considered computation independent.
	But if it is part of generated software system, it is considered to be a portion of the implementation details.
	
\end{itemize}

Those three levels can be wrapped by two additional levels.
\cite{jeary_embedding_2011} 3 defines the level of \textit{Requirements Engineering}, which is the most abstract level and code level which is the lowest level of abstraction.
By \textit{lowest level}, the author references the lowest human intractable level of code, which can be an object oriented language like Java for example.
This results in the following hierarchy, where 1 represents the highest level of abstraction and 5 the lowest:

\begin{enumerate}
	\item Requirements Engineering
	\item Computation Independent Model
	\item Platform Independent Model
	\item Platform Specific Model
	\item Code
\end{enumerate}

\section{Model-Driven Development}

Models can be used in a software development process (\ac{MDSD}) or as part of a software architecture (\ac{MDA}).
Software is also a model, which represents real life objects and procedures.
Since it is well structured by default, it is fairly easy to directly create models from an existing code basis.
\ac{UML} is a widely used modeling languages, which can represent static, dynamic and interactive aspects of software systems.

The problem is, that the domain specific structure of the code can be hard to understand, because the abstraction level of a modern object oriented programming language is much lower, than the domain requires.
\MakeUppercase{Völter} and \MakeUppercase{Stahl} describe the result of scattering relevant information in the code as:
\textquote{The much-treasured inner structure is present in a cloudy, distributed, and of course also a strongly individualized form.}\footnote{\cite{stahl_model-driven_2006} 14}
That means, that the domain's structure, components and behavior definition is represented in the software code, but in a way that is hard to understand.
That is, because the relevant parts and the domain critical definitions are spread in the software's code.
It is also harder to unterstand a domains properties, by only reading the programming language, because platform and domain code blend together.

A software model, like an \ac{UML} class diagram are an old tool to document and model software systems.
The problem here is, that the abstraction level of an \ac{UML} class diagram is often times the same as the programming language itself.
A developer could find himself transforming a class into a diagram or generating a diagram, which does not contribute to the understanding of the domain structure.
This happens, because the abstraction level of a modern programming language and that of an \ac{UML} class diagram are similar.

To address the problem of widely spread domain implementations, the program's relevant business logic is compressed into a model, which defines and represents the program on an abstract and formal level.
The model can be a textual \ac{DSL}, a graphical diagram or even an informal text, that is then converted into a concrete syntax.
It contains all the domain relevant information, which is required to derive a full implementation.

This procedure results in a model becoming part of the source of a program, and not only the documentation.
A good model syntax can lead to the model being the documentation itself.

\subsection{Modeling a Software System}
\label{sec:modelingasoftwaresystem}

An existing implementation can be analyzed, to create a model from it.
If no model infrastructure exists, this is the way to approach the transformation.
The existing implementation is a software project, that implements business logic and platform depended interaction using a native programming language and frameworks.

\begin{figure}[h]
	\centering{
		\includegraphics[width=0.8\linewidth]{images/background/mdsd1.pdf}
	}
	\caption{Analyzing an existing implementation (according to \cite{stahl_model-driven_2006} 15).}
	\label{fig:mdsd1}
\end{figure}

Figure \ref{fig:mdsd1} shows, how the code basis of an existing implementation of a software system can be split into three categories\footnote{\cite{stahl_model-driven_2006} 15}.

The \textbf{schematic repetitive code} is the part, that needs to be abstracted into a model.
It contains the application specific business logic, that is later modeled by a user.
It does not contain code, that is equal for all applications in the same domain.
A real world example are controller implementations of a web service.
Since they implement application specific business logic, but follow the schematic repetitive design paradigm of service controllers, this part of the code is called \textit{schematic repetitive}.

\textbf{Generic code} is derived from the domain and target platform and not from the concrete application instance.
Therefore, it is not modeled, but generated by the domain's \ac{MDD} infrastructure.
It is mostly boilerplate code, that should not be part of the model.
It is the same for every application and is later part of the generator templates.
Some of the generic code can be handled by classic techniques like object oriented programming and libraries.
But the integration of a library in all of the relevant places is still generic code, that is not relevant for the business logic.
A real world example is an image cache, that downloads and caches images using a cache policy.
This cache implementation must be used for every image in the application.
A generator can generalize the use of images and generates the use of the cache for every image that has been modeled.

\textbf{Individual code} is unable to be represented by the model, because it is application specific code, that cannot be generalized.
This happens if the code is not used in other applications and the model should not support those features, because of design decisions to keep the model as generic as possible.
The tools need to make sure that this code can be integrated into the generated code later.
To simplify the \ac{MDSD} process, individual code should be avoided as far as possible, because in the worst case the individual code makes up all of the application code, which eliminates the possibility to use a model to represent the application.
In the best case, the individual code part is empty.
This allows for a more simpler tools, because the application can be generated completely out of the model and there is no need for individual code interfaces.


\begin{figure}[h]
	\centering{
		\includegraphics[width=0.8\linewidth]{images/background/mdsd2.pdf}
	}
	\caption{Separation of code into source code and generator code (according to \cite{stahl_model-driven_2006} 15).}
	\label{fig:mdsd2}
\end{figure}

Figure \ref{fig:mdsd2} shows, how the analyzed parts can now be separated into source code and generator code.
Source code describes the part of the code that is application specific and is developed by the \ac{MDD} user.
The application model and individual code should be checked into source control, since that is the actual source of the application, given a \ac{MDD} infrastructure exists.

Platform and domain code are part of the \ac{MDSD} tools.
It is baked into the tools, so it is part of the output while generating.
In case of code generators, this code would be part of the templates.
In theory, it is now possible to develop any app, that fits into the domain, by only defining the model and the individual code.

\subsection{Meta Programming}

Now that there is a model of an application, a concrete implementation has to be created from the model.
Code generators can take the model and generate a software system, that is an instance if the model.
Another approach are model interpreters, which execute the model directly, similar to a virtual machine.
Code generation enables capabilities, that are available to the generated code.
By using meta programming, more concepts can be expressed than the target language allows.

\subsubsection{Code efficiency}

A code generator has the potential to build highly efficient code, by using the fastest statements, which have the fewest overhead.
Modern programming language features like flexible syntax, polymorphism or reflection might cause the application to run slower then needed\footnote{according to \cite{stahl_model-driven_2006} 181}.
A code generator can perform polymorphism and reflection on generation time and generate direct code that runs fast.
The flexibility now lies in the model and not in the generated code.
This means, that the generated code is not supposed to be maintainable, because it is highly optimized low level code.
Instead, the code generator and templates are subject to maintenance.

Code size can also be reduced, by only creating the needed parts.

\subsubsection{Platform language restrictions}

Is is possible to break out of the target language's restrictions.
The level of abstraction of the meta model and its defined language does not need to match that of the target language.
A code generator can generate code that reflects higher level of features.
It is possible to generate code that integrates the concept of \ac{OOP} into a non \ac{OOP}-language.
For example: A generator can generate structs and function pointers in a C/C++ application to imitate objects.
The generated code on it's own is not considered safe or structured, but the static context analysis that is done while creating the model performs the sames checks as an \ac{OOP}-compiler.
Another example are generics, that can be used by generating untyped objects\footnote{$Object$ (Java), $void*$ (C/C++), $any$ (JavaScript)} and perform a static context analysis while generating the code.
Any feature that is known from modern languages can be used by imitating the compiler and its static context analysis.
In a more extreme case, a virtual machine can be integrated into the application which can run any code.
Another example are code decorators, that can easily be supported by generating additional code to any structure, although the target language does not support decorators. 

\subsubsection{Portability}

The higher the abstraction level of the meta model, the more platforms can be supported.
The static context analysis is independent to the platform and should not integrate platform specific restrictions.
It is possible to overcome those restrictions as described in the previous section, but the development overhead could increase massively.
Therefor it is easier to reconcile platform and model capabilities in a clever way.
A transition to another platform or supporting multiple platforms at once is possible and is one of the core strengths of \ac{MDD}.

\subsection{Code Generation}
\label{sec:generation}

Now that a model exists, code can be generated out of it.
Before code can be generated, the model has to be transformed into a generation oriented model.
Its purpose is the simplification of the generation process by providing the target structure, which then can be converted to text directly.
The transformed model is an model instance, that is targeting a specific platform.
If the model's abstraction level is low and close to the target platforms, the transformation is not needed.

A typical example is a high level model, which could be implemented in any imperative language, would be transformed into a \ac{UML} class model.
Object oriented programming code can then be generated out of the class model in a trivial, non challenging way.
There are several techniques to create program code from a transformed model.
The two most popular ones\footnote{according to \cite{stahl_model-driven_2006} 186} are templates- and \ac{API}-based code generators.

\subsubsection{Template Based Generators}

A template is a text file, that contains the target source code combined with complex placeholders, which can iterate over data.
A template mixes the concrete target code presets with a markup notation.
The handlebar notation is one of the simplest form of template based code generation.
A well formed model, which is considered the input is applied to a template, which results in an output text file.

The following example generates \ac{HTML} code by using a \ac{XSLT}-based templates and \ac{XML} models.
As described in listing \ref{lst:xsltmodel}, the model represents a list of text lines.

\begin{lstlisting}[caption={data.xml - The model that is data.}, language=html, label=lst:xsltmodel]
<list>
  <entry>A</entry>
  <entry>B</entry>
</list>
\end{lstlisting}

The template is displayed in listing \ref{lst:xslttemplate}.
It defines, how the data is transformed into \ac{HTML}.
The template is correct \ac{XHTML} by itself.
This way, a structured and well-defined transformation can be modeled.

\begin{lstlisting}[caption={template.xslt - The templates that transforms data into HTML.}, language=html, label=lst:xslttemplate]
<xsl:stylesheet version="1.0" xmlns:xsl="http://www.w3.org/1999/XSL/Transform">
	<xsl:template match="/list">
		<ul>
			<xsl:apply-templates select="entry"/>
		</ul>
	</xsl:template>
	<xsl:template match="entry">
		<li>
			<xsl:value-of select="." />
		</li>
	</xsl:template>
</xsl:stylesheet>
\end{lstlisting}

A \ac{PHP} script can now trigger the \ac{XSLT} processor and generate an output file.
This output file is shown in listing \ref{lst:xsltoutput}.

\begin{lstlisting}[caption={XSLT ouput.}, language=html, label=lst:xsltoutput]
<ul>
	<li>A</li>
	<li>B</li>
</ul>
\end{lstlisting}

The mos interpreter uses \ac{XSLT} to generate the mandatory license \ac{HTML} page, which is based on a configuration file.
Instead of the full license text, every third party license reference only contains the license type identifier (e.g. MIT).
This way, the source file is easy to maintain and safe to extend.

This concrete usage example of \ac{XSLT} yields several benefits:

\begin{enumerate}
	\item Changing the style and layout of the HTML file requires only a few changes. Adding or changing the \ac{CSS} classes of elements only requires to change the one occurrence in the template.
	\item The \ac{URL} of an entry was displayed as the text content of an $a$ element, but also occurs in the $href$ attribute. Changing the \ac{URL} later, can cause inconsistencies, if one of the two occurrences are forgotten.
	\item Instead of copying the license text to each entry, only the identifier (e.g. $MIT$) has to be defined.
	The template appends the correct text automatically.
	\item Defining a template and a model of this small size actually made the handling more easy and future changes can be made much faster, while keeping the file consistent.
\end{enumerate}

In larger projects, the template can become very complex.
Using intermediate models as described in section \ref{sec:generation} which are generated from the source model, can reduce the complexity, branching and nesting of the template, because the intermediate model has either the same or a correspondingly compatible structure.
Nevertheless, templates will get very big very fast and are bound to the structure of the target platform.
In \cite{stahl_model-driven_2006} this approach is called ''Templates and Metamodel'', while \textit{Metamodel} is the intermediate model, that is used to generate code directly.

\subsubsection{API-based Generators}

The downside of templates are their untyped and unsafe character.
It is possible to make small syntactic errors, that are unsafe and could cause the result to fail building.
Especially in interpreted target languages like \ac{PHP} or \ac{HTML} the error only occurs when the respective line is parsed or ran.
Since the template elements are scattered in the file and critical segments are hard to make out, this approach might be unsuitable for large or complex projects.

An \ac{API}-based generator is a programming library for a development language, which provides an interface to the \ac{AST}.
It's purpose is to build up an \ac{AST} using a type safe language like Java.
Now the generator can be structured with Java's language features like Polymorphism, Interfaces and inheritance.
The \ac{API} makes sure, that only syntactically correct code is produced.

\ac{API}-based generators are direct alternatives to templates and are examined by \cite{stahl_model-driven_2006} 192.
They are described to be the most popular type of code generators.

It is also possible to generate code for different target languages, which use the same \ac{AST}.
For example, it is possible to generate Kotlin instead of Java code, since Java is a subset of Kotlin.
Also, the .NET runtime makes it possible to generate Visual Basic, C\# and C++ code from one \ac{AST}.

\begin{lstlisting}[caption={Using an \ac{API} to generate Java code, written in Swift}, style=swift, label=lst:javagenerator]
let personClass = ClassDeclaration(named: "Person");
...
let studentClass = ClassDeclaration(named: "Student");
studentClass.baseType = personClass
personClass.typeAttributes = .public
\end{lstlisting}

\begin{lstlisting}[caption={Java code result of lising \ref{lst:javagenerator}}, style=myjava, label=lst:generatredjava]
public class Student extends Person {
}
\end{lstlisting}

Listing \ref{lst:javagenerator} shows an example usage of an \ac{AST}-based Java code generator, which is written in Java itself.
This form of meta programming allows the creation of any valid code by using an object oriented interface.
Since it is independent from any textual representation of the language, syntax errors in the generated code are impossible and any other language with an compatible \ac{AST} can be generated.

If the development and target platform are the same, the \ac{AST} framework could make use of reflection to provide a more safe interface.
For example, it could be possible to reference existing Java classes not by it's string name, but rather by using direct type references, like $System.Object.class$.
It might limit the set of compatible target languages, but the example of Java, Kotlin is still possible.

Using jQuery to manipulate or build \ac{HTML} content is also a form of \ac{API}-based code generators, since jQuery provides a safe\footnote{Not all jQuery programming errors can be checked on compile time, despite a typed interface using TypeScript. Syntax errors in the generated code are not possible.} interface to the \ac{DOM}. The generated \ac{HTML} code can then be read from the jQuery \ac{AST} object.

	\centering{
		\includegraphics[width=\linewidth]{images/background/apibasedgenerator.pdf}
	}
	\caption{API based code generator infrastructure, derived from \cite{stahl_model-driven_2006} 192}
	\label{fig:apibasedgenerator}
\end{figure}

Figure \ref{fig:apibasedgenerator} illustrates the infrastructure of an \ac{API}-based code generator.
The darker rectangles (\ac{API} and \ac{AST}) represent components, that are independent to the domain, but are directly derived from the target platform.
That means, that pre-existing solutions\footnote{E. g. CodeDOM for .NET Applications \url{https://docs.microsoft.com/en-us/dotnet/framework/reflection-and-codedom/using-the-codedom}} can be used.

This approach is suitable for large projects, that need a lot of generated code.
If most of the generated code is boilerplate code, which has only few injected lines, templates might be the better approach.
Since templates represent the resulting structure, they might be easy to read.
\ac{API} generators are more maintainable since changes are type safe and less sensitive to errors.
The initial expense could be higher in comparison to templates.
It is possible to mix these approaches by generating parts independently.

\subsection {Model Interpreters}

Instead of generating code, it is possible to interpret and run the model directly, similar to a virtual machine.
The model is parsed by an interpreter, that uses the meta model to define an \ac{AST}, which is the internal representation of a model.
This representation can now be transformed to improve handling.
It is possible to generate internal code out of the model which then is interpreted.
Examples for that are Java or JavaScript, which are converted to byte code that is then ran in a virtual machine.
The actual interpreter of the byte code is independent to the original \ac{AST}, so a more modular infrastructure is possible.
Another approach is to run directly on the internal data structure of the model.
This methods seems easier at first, but ties the run time directly to the \ac{AST} which makes maintaining and introducing changes harder.

\begin{figure}[h]
	\centering{
		\includegraphics[width=\linewidth]{images/background/interpreter.pdf}
	}
	\caption{Interpreter infrastructure for common code interpreters}
	\label{fig:interpreter}
\end{figure}

Figure \ref{fig:interpreter} illustrates a typical interpreter that is used for \textit{JavaScript} or \textit{Python}.
A model interpreter can use a similar approach, by transforming a model into a domain specific byte code or traversing the \ac{AST} directly.
To decouple the meta model from the interpreter, a intermediate data structure can be created by transforming the model.
That way, a more modular infrastructure is achievable.
Also, the transformed data structure can be optimized for speed by evaluating it's internal relations and constraints before it's is run.
This will improve execution speed of repeating code like recursion or iteration.
The approach of how to run the model is then directly dependent on the structure and therefore the domain.
Also, the different approaches can each have specific levels of abstractions.
The \textit{V8 JavaScript} engine runs on a very low level of abstraction, since assembly-like byte code is interpreted.
The interpreter that is discussed in future sections, \ac{mos}, performs on a higher level, since it is executing user oriented tasks like sending \ac{HTTP} requests or displaying \ac{GUI} elements with data.

An interpreter is dynamic and can swap or extend the current model.
In the domain of web pages, JavaScript can be loaded on demand without recompilation.
In the domain of mobile applications, the model can be changed or replaced without updating the app itself.

\subsection{Teams in MDD}

There are new additional roles in a team, that is doing model-driven development. \textsc{Stahl} and \textsc{Völter} \cite{stahl_model-driven_2006} describe eight roles, that will be relevant to the process.
The roles allow a distinct management of workforce, which results in not needing everyone to have broad and deep knowledge.
While classical application development requires the developers to have deep knowledge of the platform and the domain requirements, model-driven development can separate and re-connect the different levels of knowledge.
The following section will describe the roles of model-driven development teams in detail.

\begin{enumerate}
	\item \textbf{Domain Analyst}
	
	The domain analysis evaluates the similarities and differences within a domain.
	The goal is to compile a list of requirements, that is equal for every product.
	These requirement will later be unified by the code generator or model interpreter.
	This analysis can take up multiple days and requires iterative workshops with various team members like product managers and domain architects\footnote{\cite{stahl_model-driven_2006} 380}.
	
	\item \textbf{Language Designer}
	
	As a link between the domain expert and the domain architect, the language designer defines the \ac{DSL} and implements reference models.
	The results of the domain analysis directly incorporates into the \ac{DSL}'s design.
	
	\item \textbf{Domain Architect}
	
	The \ac{DSL} now has to be implemented and the domain infrastructure has to be developed.
	This includes language tools, model transformers, code generators and model interpreters.
	
\end{enumerate}

Each of those roles can be split up into sub roles, that will be responsible for more granular tasks.
Depending on team size, roles can be taken by more than one person and one person can take multiple roles.
In addition, classic roles like product managers, test engineers and customers are also present in a model-driven development approach.
