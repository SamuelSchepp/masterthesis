\chapter{Concept}
\label{chap:concept}

This chapter will describe the goals of \ac{mos} and it's iOS implementation.
The first section will describe the informal requirements to the model.
Also, the model's evolution and progression is discussed, which is relevant to the main idea behind the model interpreter.

\section{Implementation Goals}
\label{sec:implementationgloas}

The main problem of mobile app development is the iteration process, which consists of collecting requirements, creating conceptual prototypes and implementing on the two main platforms.
Since requirements are informal, incomprehensive or change during the development, multiple iterations have to be done to converge to the final product.

To minimize implementation times and be able to do more iterations, \textit{Same Day Delivery}\footnote{The term originates from parcel shipping and means that a parcel arrives at the customer on the day of the order.} tries to eliminate the times, that is independent to the customer.
That means together with the customer the app is designed and the model is created in one session.
Since the functional scope is already defined and all of the feature are implemented in the interpreter, the domain expert who is directing the meeting can create the model live and on the fly.
The App can be tested by the customer directly, without the need to compile an custom app.
Common tasks like downloading data from a web source, forms and user interaction are configured on a high level, so the customer is able to understand aspects of the app.

\subsection{Functional Model Requirements}

To be able to validate the meta model and an implementation, requirements have to be defined.
A concrete model also has requirements, which are application specific however.
That's why, only the meta model and the implementation are considered.
Since this chapter is about the concept of the \ac{mos} interpreter, the discussion on the requirements of the implementation are found in chapter \ref{chap:implementation}.

The following paragraph lists the functional requirements to the meta model.
Those are aspects that have significantly influenced design choices.
All of the listed requirements are high level ones, that reflect the paradigm.
However, the number of actual concrete requirements would be too high to be presented here.

\begin{enumerate}
	\item \textbf{Separation into independent sub-models}
	
	An app loads a root node, which then can navigate to a different node, or show it's preview.
	Every node is theoretically independent and can be used as root node.
	To enable interaction between nodes, an action that displays a new node can inject additional data into the node.
	An example is a master-detail-view, which is a common pattern in application design.
	A master view is a list of all objects.
	It knows everything and is connected to the database.
	When the user clicks on one of the objects, the detail view is presented by passing either the database ID or the whole object as parameter.
	The same approach is possible with nodes, which results in the detail view node not being independent to the master view node.
	
	\item \textbf{Fully customizable layouts}
	
	A layout can be defined by arranging different kinds of layers by configuring layout constraints. Also, native navigation bar, tab bars and side menus can be set independent to the layout.
	All of the common layer types are supported like text, image and color.
	Those layers have layer parameters that make them customizable.
	Position and size can be defined with constraints, by binding the edges of a layer to the edges of other layers or the superior container.
	The binding can be modified with additional margins so that dynamic and portable layouts can be defined.
	The system is inspired by Apple's Auto Layout\footnote{Auto Layout calculates absolute positions and sizes based on constraints between layers. \url{https://developer.apple.com/library/archive/documentation/UserExperience/Conceptual/AutolayoutPG/index.html}}.
	
	\item \textbf{Declarative and reactive behavior}
	
	Everything that is depended on something else needs to update on changes.
	For example: If a layer is depended on an \ac{SQL} query and the corresponding tables change their contents, the layer needs to update itself with the new data.
	Another example are layers that have a form data key and thereupon synchronize their contents with the view's form.
	Layers can also have a setting data key, which enables a layer to synchronize it's value with a field in the database.
	This makes it possible to use the value in \ac{SQL} queries, apply it to templates or uploading data to a service.
	
	An example that enables a React\footnote{Reactive and functional UI framework. \url{https://reactjs.org/}}-like behavior is the \textit{layout layer}.
	It's parameters are an \ac{SQL} query, a layout name and a template name.
	The layer runs the query and applies the data to the template, and the template to the layout and displays the layout.
	If one of the tables that are selected in the query changes, the \textit{layout layer} re-renders itself.
	
	\item \textbf{Separation of data, templates and layouts}
	
	A template defines layer parameters by taking a dictionary of data and applies it to itself. The result can then be applied to a layout.
	A template overrides layer parameters.
	String literals inside the template can contain mustache\footnote{The string \textit{\{\{data.field\}\}} is replaced with the corresponding value from an object. \url{https://mustache.github.io/}} handles, which are evaluated against the data.
	This allows reuse components and enables dynamic decision making during run time.
	A layout-layer can dynamically load a sub-layout and apply a template.
	Since the name of the template is part of the layout-layer's parameters, it can be evaluated on run time and be modified by a mustache syntax.
	
	\item \textbf{Database definitions}
	
	An \ac{SQL} database can be designed, that is in sync with a data source.
	Fetch intervals, queries and structure can be defined.
	Column based queries for inserting data from a data source enables a structure, that is different from the source.
	
	\item \textbf{Actions}
	
	An action can perform common tasks like uploading data, setting layer values, or triggering fetches.
	Actions can be fired by clicking on a layer or are triggered by other actions, which take the result of the previous action as context data to use.
	The action's parameters can be customized by using handlebar notation and injecting context specific data into the actions.
\end{enumerate}

\subsection{Non-Functional Model Requirements}

The following paragraph lists the non-functional requirements.

\begin{enumerate}
	\item \textbf{Easy to understand}
	
	Existing models are easy to understand.
	Names are clear and there is no room for interpretation.
	A model developer has to be able to understand an unknown app's structure and perform changes.
	
	\item \textbf{No boilerplate code}
	
	All of the configuration contains highly concentrated information and no boiler plate code has to be defined.
	Similar configurations can be combined to one reusable resource\footnote{Layouts, Templates and Nodes can be reused.}.
	
	\item \textbf{Easy to parse by an interpreter}
	
	The model is easy to parse and is encoded in a common format.
	Validators can check for simple errors.
	While parsing performance is not the issue, the interpreter's code base needs to be as simple as possible to avoid errors and to be maintainable.
	
	\item \textbf{Backward and future compatibility}
	
	Adding new functionality has to be added without any old interpreters to fail on new models and a new interpreter to fail on old models.
	Also, every new feature as to be developed with the future in mind.
	The feature has to add important value to the models capabilities and has to fit into the design principles of the current feature set.
\end{enumerate}

All of the above functional and non-functional requirements are developed both with a bottom-up and a top-down approach.
Since the interpreter contains most of the business logic and implicitly defines the meta model, it is one of the key aspects.
That is why, the meta model is structured to be parsed and interpreted in a simple and direct way.
In the same way, the usability of the models are also considered, so that both sides are balanced.


\section{Model Syntax}

To meet the majority of the non-functional requirements, \ac{JSON} is used to define the model.
It allows the client to parse common structures and define optional and default values in an maintainable way.
Also, since the model uses declarative paradigms, C-like imperative language are not suitable.

The model is divided into sub-models.
The first model is called root node and is loaded by the interpreter on the first app start.
The \ac{URL} to the root node is compiled into the app.
A node defines some of the meta data, static data, a starting layout reference and events.
The meta data contains an \ac{URL}, called \textit{Update URL}, that is used by the client to synchronize layouts, templates and database definitions.
Database definitions also contain \acp{URL} that are requested if needed to synchronize actual content data.

Next to node definitions, template, layout and database configuration files are part of the model.
The requesting the root node from a server will also append the additional files, that are cached by the client.

The client implementation is able to optimize the data usage by only reloading data that has changed.
\ac{HTTP} techniques like ''etag''\footnote{Entity Tag, a unique identifier assigned by the web server to specify a resource and it's version.}, ''modified-since''\footnote{A \ac{HTTP} header containing a timestamp.} or delta updates\footnote{Requesting a resource by \textit{/api/file.60.json} will deliver the changes of the last 60 minutes.} are used to optimize the update process.
The model's job is to interconnect web services and the client.
Optimizations can be implemented without changing the model, since they are not coded into the model in any way.




\subsection{Drawbacks}

\ac{JSON} as a model syntax leaves a lot of room for interpretation, since the semantic and syntax are strictly separated.
A \ac{DSL} that is parsed by an compiler has semantics integrated into the compilation or parsing process, which allows finding errors much earlier.
A common example is the parsing of a expression tree, which uses a top down parser, that parses parentheses.
Depending on how the parser is constructed, it allows incorrect parentheses or rejects them because of a parser error.
Defining the same expression as a \ac{JSON} object 

This issue can be fixed by using \ac{JSON} Schema\footnote{\url{https://json-schema.org/}}.
A schema is a \ac{JSON} object which describes the syntax of given \ac{JSON} data.
It is possible to define required keys, possible values and other rules, that could help the model creator to find error faster.
Also, a static context analysis could be used to catch even more errors.
There are many occurrences, where layer, query and other objects are referenced.
Those could be checked statically or auto-completed in the editor.
At the moment, these tools are not implemented and typical errors occur regularly.
The problem is, that those tools take a lot of time to develop, while they do not add any new functionality to the model or the interpreter.

The systems limits it's functionality and new features are hard to implement, since they have to be designed to work together with all of the existing features.
This means, that the reason of why something is not possible to implement at the moment are incomprehensive and the customer cannot understand, why new features are much more expensive.
An example would be custom user interface animations.
While it is easy to hard-code some animations to specific elements in a classical app, it is much harder to define a representation for the model.
That is because animations then have to be valid on any scenario.
If an animation looks weird, there has to be a decision if the model is badly designed, or there is a error in the client code.
Also, the level of abstraction has to be defined so all of the future animations can be expressed and the way of defining the animation must be compatible with the different platforms.

\subsection{Iterative Development Process}


The functionality of the model and it's interpreter is constantly evolving.
Layers and actions can be extended easily and new features can be added really fast.
The development method that is used in the example of mos is an agile iterative development approach.
A new feature needs to be implemented into the platforms and a model has to be defined, that demonstrates it's aspects.

Most of the time, new features emerge because an app that is in development needs a feature, that is currently unavailable.
The development of new functionality is avoided by first trying to implement the app with the existing syntax.
For example, instead of developing highly specialized layers, the existing layers can be configured to build a new one.
This approach is used by \ac{UI} frameworks to reuse existing code or by \ac{RISC} processors to reduce the chip's complexity.

The syntax is developed top-down and bottom-up.
To be able to have a systematic approach to model driven development, the simplicity of the syntax is one of the key non-functional requirements.
On the other hand, a simple interpreter is also important, so some syntax elements are built to fit the current interpreter implementations.
That is why, the system is developed by considering all levels and iterative development is used.

\subsection{App Configuration}

Mobile applications for Android and iOS need to compile application specific information into the app.
This meta data can not be loaded in run time, because the operating system needs them to manage the apps.
Examples are the build number, version string, interface orientations and the root \ac{URL}.
This configuration is not fully platform independent, since the different systems require different information.
In the case of iOS, the development tool Xcode, which is required to build iOS apps will not build an app until the needed meta data is provided.

The \textit{App Config} also contains media like the app icon and the splash screen image.
Also, the \textit{Pre-Load Bundle} is included, which is a directory and contains images and nodes.
This directory is used to load data from disk instead of the network.


\subsection{App Template}

The \textit{App Template} is a template file structure that is used to create a run-able app.
On its own, the \textit{App Template} is an incomplete app container that contains handlebars placeholder.
A grunt task takes the \textit{App Config} and replaces the placeholders.
This way, an app container is created, which is able to compile and ready to be submitted to the store.
A continuous integration environment should be able to build all apps, given a list of all app names.

\subsection{App Bundle}

The app bundle is a full-fledged application, that is an instance of a specific app model.
It is the result of compiling the \textit{App Config}, the \textit{App Template} and the \textit{Interpreter Framework}.
It does not contain the model itself, because it is loaded from the web the first time the user opens the app.
This way, the model can be changed on the fly, while the app itself is still waiting for review or ready to release.

The model is interpreted by the Bundle, which lets the model run inside a virtual environment.
The \ac{mos} run time enables declarative model design and handles the data flow.
Similar to a virtual machine, the model can access subsystems and interfaces that are normally not available in the target platform.
For example, a custom event system handles app wide events and actions.
Also it provides the interface to the custom layers.
Those are layers in the view hierarchy that have more functionality then the default ones.
The biggest aspect is the calculation of changes.
Since relations are defined declarative, changes to the data have to update and re-layout the layer accordingly if needed.
Also the update of database data and the scheme is a big part of the system.

\subsection{Functional Interpreter Requirements}

The following section will list the functional requirements of the interpreter.
The functionality of the platform that is described by the model is omitted due to its comprehensiveness.
Instead other requirements will be listed, that reflect some of the design choices of the platform.
The behaviors, that are discussed per requirement are not expressed by the model, but included in the interpreter.

\begin{enumerate}
	\item \textbf{Reflects changes to the model}
	
	After the configuration changes, the interpreter app loads the changes and updates it's behavior. This includes database configurations, layouts and caches.
	
	\item \textbf{Offline capable}
	
	All of the content that has been displayed in the app must be cached and available for offline use.
	This includes nodes, images, layouts and templates.
	
	\item \textbf{Cache-first policy}
	
	All nodes are loaded from cache.
	If a node is not available, the node will be downloaded from the network.
	If the not is available but is expired, the new version will be downloaded.
	Until the download finishes, the old node will be displayed.
	
	\item \textbf{Clear cache and data}
	
	The user must be able to clear the cache.
	This way, some troubleshooting and customer support is possible.
	It may happen that a faulty node is cached that is not compatible with the other nodes, or is not configured with a correct cache lifetime.
	To prevent the need of re-install, a clear cache action is available that the user can execute himself.
	
	\item \textbf{Poor configurations lead to fall-backs}
	
	Dynamic model errors must lead to safe fall-back behaviors.
	Under no circumstances should the app crash or display incomprehensible error messages.
	Since there is no static model analysis, the app must be prepared with default values.
\end{enumerate}

Some of the special behavior of the interpreter is available to the model to be described.
The caching policies of a node can be configured in detail.
Also, the cache can be invalidated on the basis of a regular expression.
Those are a few examples of the model and the interpreter 

\subsection{Non-Functional Interpreter Requirements}

The following section will list non-functional requirements of the interpreter.

\begin{enumerate}
	\item \textbf{Low battery usage}
	
	Using the app must feel as lightweight as possible.
	That is why, the battery usage must be kept low.
	Since all of the modern mobile operating system have battery management tool integrated into the system, users can see which apps use a lot of battery.
	If the app is listed there, the probability is high that the user will consider deleting it.
	
	\item \textbf{Fast app start}
	
	The must start as fast as possible.
	Especially the first start is important, because it's the first impression that a user gets from the app.
	Because of a strict cache policy, other starts are faster anyway.
	
	\item \textbf{High \ac{UI} and animation performance}
	
	The \ac{UI} should always be responsive and adequate loading indicators must be used.
	It is accepted, that poorly designed models can lead to slow UI.
	This is possible because of the high level of detail of the models.
	
	\item \textbf{Low network usage}
	
	The app should not load redundant data or load the same data multiple times.
	Client and server-side strategies can be used.
	
	\item \textbf{Small download size}
	
	The size of the app at the time the user downloads it from the app store must be as small as possible.
	
	\item \textbf{Small app size on disk}
	
	The app should not accumulate to much data, while in use.
	New cache entries must always override old ones.
	
	
\end{enumerate}


\subsection{Implementation Weak Points}
\label{sec:implementationweakpoints}

The given interpreter client has a number of technical limitations and inconsistencies.
The following subsection will outline some of the major issues, that are part of the re-engineering process.
Most of them are non-functional requirements, that are met while the app was in development and the feature set was smaller.
While the app grew, conditions and the correlations between components became more complex.
There have been two main problems, that can be subject to a major refactorization.


\begin{enumerate}
	
	\item \textbf{\ac{UI} elements are re-painted to often.}
	
	Every change to a layer results in a re-layout and a full repaint of the parent container.
	If the layout frame did not change, a re-paint is not needed.
	Complex layouts like search results in combination with per entry conditions result in a slow \ac{UI} with up to two seconds wait time between rendering frames.
	
	\item \textbf{Every operation runs on the main thread.}
	
	To prevent race conditions and visual noise, the entire application run only on the main thread.
	Complex layouting and database-driven \ac{UI} elements block the UI thread.
	On fast devices, this works well, because there no visual noise.
	That means that the user doesn't see how the UI is getting built, but instead sees the full \ac{UI} after one display refresh frame.
	
	\item \textbf{No abstraction between native platform code and independent \ac{mos} behavior.}
	
	The app uses native interfaces like the view stack to organize it's contents.
	Instead of using a virtual node stack, the view hierarchy is traversed to find nodes.
	Also, every mos element manages or has a reference to the native result.
	
\end{enumerate}

\section{Native Approach}
\label{sec:nativeapproach}

One re-engineering approach is to fully combine native elements with \ac{mos} behavior.
The main reason for this approach is the simplicity of the code and the low distance to the native components.
This approach works for a rather small interpreter feature set.
To compensate bad performance, simple and highly optimized code should be used.
The code is easy to understand and simple optimizations can be applied.

The idea is, that native components are extended with mos behavior.
That means, that everything happens directly on the main thread and on native \ac{UI} elements.

One key element is a centralized change management system, which tries to collect all changes that happen within one user interaction.
The changes are then combined to reflect one resulting change.
The problem is, that applying changes to the native elements can result in new changes.
If a layer is configured to change it's state based on the child count of another layer, the \ac{UI} has to be rendered multiple times, which is very expensive.

While the feature set grew, the requirements to the system got higher.
New features like the application of changes to layers in a different node result in a high workload on the main thread.
Introducing concurrency now is only possible on a few marked-out aspects, because the code operates on the \ac{UI} in an unstructured way.
\ac{UI} operations are only allowed on the main thread, otherwise, the change will be ignored.
The opposite of this approach is explained in the following section, which tries to separate \ac{mos} and \ac{UI} code.

\section{Virtual Machine Approach}
\label{sec:virtualmachineapproach}

\textsc{Popek} and \textsc{Goldberg} \cite{popek_formal_1974} defined 1974, that a virtual machine is a duplicate of a real computer.
But as programming languages evolve and virtualization allows more complex systems, a virtual machine does not need to have real correspondence, but can rather implement any runtime system.
Popular virtual machines, that are not based on real computers are the \ac{JVM} or the V8 JavaScript engine.

As described in section \ref{sec:implementationweakpoints}, the current iOS implementation is hard to maintain and has performance issues, because native code and mos behavior are mixed.
There are only few specific conditions under which performance optimizations by concurrency are possible.

To improve performance and maintainability of a model driven mobile application, the approach of virtual machines could be used.
The \ac{VM} is a software system that is implementing the platform specific behavior.
In the context of \ac{mos} that would be conditions, states, layouting, data and caching.
It could be integrated as an external framework and is independent of the actual platform and user interface.
The system would run in a own thread and not on the main thread.

\subsection{Native Components}

Every interface to the native \acp{API} are managed by custom, platform depended components, that can be registered in the \ac{VM}.
For example, an \ac{SQL} interface is provided that will open connections and run \ac{SQL} statements.
The \ac{UI} component will be asked to render some layers.
All the layers are already lay-outed and their respective sub-layers evaluated.
It should be as easy for the native components to perform their work, since they only translate the system's objects and tasks to native ones.
The logic between elements will be handled by the virtual machine.

Since the system will run on it's own thread, the \ac{UI} will be fast and responsive and only \ac{UI} related work will block the main thread and with that the \ac{UI}.

The implementation of the \ac{VM} logic is easier, since most parts are already implemented, but have to be extracted and refactored to not have circular dependencies.
Existing performance improvements should be evaluated if it suites the new system.
One example is the custom ContextChange pattern, that will collection all changes and will merge them to one resulting change that will be delegated to the layers.
This will be used if multiple actions will be fired.
If the database is changed several times during this time, there will only be one notifications to depended layers.
This should be used in the new \ac{VM} system to prevent multiple redundant messages to the host.

\subsection{Virtual View Hierarchy}
\label{sec:virtualviewhierarchy}

One of the major benefits is the virtual view hierarchy.
Inspired by React's virtual \ac{DOM}, the \ac{VM} handles a virtual view hierarchy, which is used to calculate layouts and the layer changes.
While it is very expensive to layout and render native view components, virtual elements can be lay-outed very fast, because no user interface render system is involved.
This means that all changes and their resulting changes can be applied on the virtual view hierarchy in multiple iterations without the need to interfere with the main thread.
Layers that have changed will be marked as dirty and after every change is processed, the actual \ac{UI} is updated to reflect the virtual hierarchy.

To minimize the amount of change application to the view, only on useful occasion
The application of changes in the native view should only happen on useful occasions, to minimize the main thread utilization.
The web \ac{UI} framework Angular uses a change detection system, that hooks into the browser specific event system to trigger an update only if changes are even possible.

\subsection{Shared Code}

It is possible to develop the \ac{VM} system as an external framework.
If it would be written in a language that can be compiled to native frameworks like KotlinNative\footnote{KotlinNative builds native C++ libraries and Objective-C frameworks from Kotlin code. \url{https://kotlinlang.org/docs/reference/native-overview.html}}, this code could be shared between iOS and Android.

The main goal is, that the virtual machine can run on any mobile operating system, implements and represents the \ac{mos} system and it's design philosophies.
A native client should provide platform-specific implementations like the \ac{UI} and the device context.

\subsection{Testability}

With the current implementation, only some isolated parts can be unit tested.
Critical parts like layouts, conditions and action cannot be tested, since they all rely on the native \ac{UI} components.
Also, the code has many circular dependencies, which makes mocking impossible.
Behavior of \ac{mos} is directly depended on native \ac{UI} elements and vice versa.

Automated UI tests are expensive to maintain and are hard to evaluate.
Using a virtual machine that runs headless\footnote{Headless is a mode of virtual machine hosts, that runs the guest system without a user interface.}, unit tests are possible since any app can be used to test against and simulate user input.
It is possible to inject virtual app models into the \ac{VM} that can be used to test against.
This allows for fast test setups and evaluation.
