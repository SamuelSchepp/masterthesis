\section{Summary}


Fabrik19 is a company, that develops marketing and e-commerce applications for digital signage and mobile apps.
To be able to update the contents, layouts and behaviors of the applications, a model interpreter infrastructure has been developed, so there is no need to update the local application bundles on the physical signage devices or the mobile applications.

The model defines layout, behavior and data.
The configuration allows the definition of complex layouts and interaction.
This makes it very powerful and functional, while still being very flexible.
If the configuration changes, the applications download the changes and update their behaviors accordingly.
The model fits into the definition of the general and specialized model theories, that are defined by Stachowiak \cite{stachowiak_allgemeine_1973} and Seidewitz \cite{seidewitz_what_2003}.

Because of low starting requirements, the interpreter's performance is noticeable worse than a native application's performance, that implements the functionality directly and natively.
This is because a software architecture has been implemented, that directly maps the models capabilities to native \ac{UI} elements.
Since changes to the \ac{UI} are happening multiple times with only one user interaction, the \ac{UI} thread is very busy.
An interaction example is a database change, that could trigger layout updates.
If the interaction produces multiple interactions, all the changes are applied to the \ac{UI}, blocking the main thread.

An alternative approach of implementing the interpreter is the virtual machine approach, which simulates the app's runtime with basic data structures.
This system is independent to any native \ac{UI} implementation and the main thread.
This means, that any change can be evaluated and the virtual \ac{UI} hierarchy can be created on custom background threads.
After the evaluation is finished and every calculation is done, the changes are applied to the actual native \ac{UI}.
The application of changes is fast, because only the resulting changes are being applied.
This means there is no duplicated work.
The idea is inspired by the virtual DOM of React Native and the event zoning of Angular.

A use value analysis illustrates, that a re-engineering would increase the overall implementation quality by 6.36\%.
This shows that a re-engineering would improve the client application only by a small margin, because the overall requirement implementation quality is high.

\section{State of the Art}

There are no direct alternatives to the mos software.
But systems that implement a similar idea exist.
React Native is a JavaScript interpreter, which runs JavaScript code using WebKit's JavaScriptCore.
The UI declaration is translated into native elements and user interaction results into a re-evaluation of the virtual view hierarchy, which is then applied to the native view.
This way, user interaction, animations are fast and native functionality is available in the JavaScript context.
The user code is bundled into a single file, which can be swapped at any time.
Expo is a React Native infrastructure and framework system, which allows the replacement of the JavaScript bundle file, when the app is started for the next time.
The \ac{UI} is defined by \ac{JSX}, which allows embedding XML like code as expressions into JavaScript.
This makes it possible to directly write \ac{HTML} style \ac{UI} code, while still being able to inject JavaScript data.

The evaluated virtual machine approach that is discussed in this thesis is implemented by React Native.
While the proposed architecture is only called virtual machine, because of it's strict separation from the main thread, the React Native implementation actually uses the JavaScript runtime environment for it's main calculation and virtual \ac{DOM} evaluation.
This is why, React Native represents the most advanced version of the virtual machine approach.
The capabilities of the React Native runtime are to high and the requirements of mos would not be met.
The model, which is the JavaScript code, is to complex, because it is not declarative, but imperative.

Another similar software system is \ac{TVML} by Apple.
Using a markup language that is similar to \ac{HTML}, apps for Apple TV can be designed that shows video content and library views.
Apple TV is a set-top box for televisions that is comparable to Amazon Fire TV.
\ac{TVML} contains predefined layouts and data interfaces to present video libraries.
Since video library apps are very similar, \ac{TVML} can be used to build a fully functional app, which fully supports the remote's gestures and system functionality.
For advanced logic, TVMLKit JS can be used, which is a full JavaScript environment with \ac{TVML} specific classes and objects.
The client application is responsible for loading and displaying the markup definitions, which is handled by classic, native code.
This means the loading and caching policies can be fine-tuned to match the requirements.
\ac{TVML} is comparable to the mos software system, because the model can be rendered server-side and it is declarative by design.

\section{Next Steps}

This section will discuss further development approaches and the steps that should now be taken by the team.
This includes alternative architectures and concrete implementation approaches.

\subsection{Continuous Reengineering}
\label{continuous-reengineering}

The next step is the actual implementation of the interpreter.
It could be considered a full rewrite, but the more intelligent solution is a continuous evolution, which transfers the application into the target architecture.
The described approach fully separates the virtual runtime and the user interface.
This radical step can be applied to the interpreter by separating all the aspects step by step.

In the beginning, the UI behavior can be extracted from the various layer implementations, by sub-classing native view elements and implement view specific behavior.
This way, the dirt flushing mechanism can be implemented step by step.
The next major problems are the view controllers, which are the windows that present the node layouts.
Those also need to be separated, so the window management is working with the virtual windows, which directly use the native elements.
After that, the dirt flush mechanism is fully implemented so that every change only results in a dirt flag, which will then be applied to the view after the interactions are finished.

These steps can be considered to be a sprints where the application is executable after each iteration.
Every change is a step in the target direction, because a good abstraction supports maintainability and extensibility.
After each change, the team can evaluated if the next step is even possible or if the architecture is not suitable at all.

The book \textit{Reengineering Patterns} \cite{demeyer_object-oriented_2002} systematically describes approaches, that support a re-engineering process of a given system.
The base patterns include, among other things, test evolution and incremental migration of sub-components.
For every approach, the author explains the intent, the problem, a solution, the respective trade offs and examples.

The tools and development processes can be applied to this section, because it directly fits into the continuous change approach which is needed to always release an updated version of an evolving system.

One example approach is the idea of coding business logic into automated tests.
One key objective of this reengineering concept is the ability to unit test domain specific logic.
Also, the book suggests to incrementally grow the test base, which is an important aspect of future reengineering processes.
The tests help to build confidence, that the current change to the system does not break it.
While not every error can be exposed by tests, they provide a basic assurance.

Another important suggestion is the continuous and incrementally migration of the system, so that it can always be deployed.
A full rewrite or parallel reengineering while also maintaining the old system results in two code branches.
This means, bug fixes and new features must be applied to both system, which can lead to increased development time and diverging functionality.

\subsection{Hybrid Approaches}

A hybrid interpreter is an architecture approach that allows native and custom code to run with an interpreted model side-by-side.
This can be achieved by embedding an interpreter into an application to offload a specific set of aspects.
An example are parts of the mobile applications of Facebook and Instagram, where some features are extracted into browser based progressive web apps\footnote{Source: \url{https://instagram-engineering.com/react-native-at-instagram-dd828a9a90c7}}.
Those run inside the app and basically behave like other native parts of the app.
These can be recognized by longer loading times and a delayed view setup.
Code only has to be written one time for any platform, the model can be loaded from network, to make it possible to change important aspects on the fly.
Promotion and news are often times models or configuration data that can be interpreted on the platform without the need to design custom views and wait for the user to update.
This approach is suitable for apps that only substitute small aspects.

Another hybrid approach is a plugin system, that is integrated into the meta model and the interpreter.
This way, the model can access native functionality that should have been compiled into the app and is then part of the interpreter.
A possible implementation could be the extension of the meta model to support calling static native methods using meta programming language features.
The plugins can access the plugin interface of the interpreter.
Depending on requirements, this interface can be very open, so the interpreter can be arbitrarily extended.
This approach can be used, if the native parts are small exceptions and most of the work is done by the interpreter.
Examples are native components in React Native or native modules in ionic.

A hybrid interpreter can also mean, that parts of the interpreter are generated.
For mos, this is partially true, because the app container is generated using a separate model.
The generator produces a native app project structure that can be used by Xcode to run and publish the app.
The app then uses the interpreter framework to display the actual app contents.

\subsection{Abstraction}

It is possible to achieve even more platform independence, by implementing the virtual machine component in a common language.
Since the idea behind the virtual machine approach is, that no platform dependent code is used, this framework could be implemented in various platforms or languages.
Table \ref{table:platformindependentabstraction} shows some examples.

\begin{table}[h]
	\begin{center}
		\begin{tabular}{l|l|l}
			Platform       & Language   & Runtime                    \\ \hline
			Native Library & C++        & Native library                 \\
			JavaScriptCore & TypeScript & JavaScript runtime environment \\
			Xamarin        & C\#        & Native library                 \\
			Kotlin Native  & Kotlin     & Native library                
		\end{tabular}
	\end{center}
	\caption{Platform independent virtual machines.}
	\label{table:platformindependentabstraction}
\end{table}

Since the virtual machine implementation needs to be a highly optimized and fast system, that runs on multiple threads, the JavaScript runtime environment is not suitable.
Xamarin and Kotlin Native introduce another layer of dependency and potential bugs.
Also, part of the overall requirements of mos is to avoid as many dependencies as possible.

The most compatible and future-proof option is C++, which can be compiled to the respective platforms and can be used directly and natively without crossing abstraction barriers.
The interface must be designed very carefully so that both platforms can work with the implementation.
C++ makes the development hard, because it is an unsafe language that must be bridged to C and then to Swift, which makes it hard for changes to be implemented.
Also, modern programming paradigms like functional or reactive programming, multi-threading, networking and JSON are not fully supported by the standard library.

Another suitable platform is Kotlin Native, which makes it possible to actually use classes and functions as if there where implemented natively.
Kotlin is a modern functional programming language with a comprehensive tooling support for documentation and unit testing.
It is a very safe language, because it supports optionals, coroutines, automatic operator synthesis\footnote{In Kotlin, data classes are collections of fields where many operators like \textit{equals}, \textit{copy} and \textit{toString} are automatically generated.} and other modern programming paradigms.
As of June 2019, Kotlin Native multi-platform libraries are still an experimental feature.

One aspect of the mos interpreter has been abstracted into a Kotlin Native framework.
A routing algorithm for the map layer has been developed as a framework so that iOS and Android can share it's code.
While iOS uses the exported framework file, the Android project includes the framework's source files and compiles them directly to Android-native machine code.

The JavaScript Core approach makes it hard to exchange data and implement callback mechanisms.
While it is easy for the iOS application to set function pointers and exchange \ac{JSON} data, the impact on the overall complexity must be evaluated by re-writing the demonstrator in JavaScript.
Since JavaScript does not allow multi-threading, this alternative should not be considered.

The ultimate goal is a common virtual machine implementation where the clients would act as user interface and platform-specific feature providers.

\section{Related Work}

This thesis elaborated two topics.
The model, it's abstraction approach and requirements have been evaluated.
In addition, the internal structure of the interpreter has been analyzed and an alternative architecture has been proposed.
The following section will collate some related papers and resources.

\textbf{Runtime Models:} The paper \textit{Systematic literature review of the objectives, techniques, kinds, and architectures of models at runtime} \cite{szvetits_systematic_2013} analyses the use of models with regard to runtime systems, by using Systematic Literature Review.
Runtime models take the idea of static artifacts one step further.
The paper's goal is to identify and classify approaches, architectures and usage of runtime models.
The research suggests problems like error localization, requirements evolution, abstraction and maintenance can be addressed by runtime models.
Those models can have various architectures, techniques and objectives.
The paper proposes those classification to improve development workflows of systems using runtime model support.

One example model is an abstract state machine that can be used to formally describe component behavior, which then can be executed directly.
The goals are abstraction, consistency, simulation, platform independence and policy checking.

\textbf{Requirements Engineering:} The paper \textit{Embedding requirements within Model-Driven Architecture} \cite{jeary_embedding_2011} deals with the topic of integrating Requirements Engineering and Business Process Management into a Model Driven Architecture project.
The objective is to improve and enhance the artifacts of requirements engineering, by using \ac{BPMN} to transform business logic into a Computation Independent Model (CIM).
This is achieved by standardizing \ac{BPMN} to be able to map the artifacts into intermediate models like the \ac{BPEL}.
Those can then be converted to \ac{MDA}'s \acp{CIM} and therefor to \acp{PSM}.

The paper concludes by emphasizing the openness of \ac{MDA}, which allows many model variations and tools that can be integrated into the process.
This way, it is possible to formally map a \ac{BPMN} to a \ac{PSM}.

\textbf{Software Architecture:} The connection between software architecture and abstraction is discussed in the paper \textit{Architecture = Abstractions over Software} \cite{jacobsen_architectureabstractions_1999}.
The main statement is that software design and architecture are a abstraction over the software domain, which elevates the architecture above the usage environment and domain.
This means, that architecture is domain independent.
The paper classifies and characterizes various aspects and implications of software architecture like the design process, tools and perspectives.
Static and dynamic aspects are also being discussed.
Dynamic aspects of an architecture include state and transitions.
